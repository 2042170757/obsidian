## 核心理念：停止检索，开始编译

### 传统 RAG 的问题

大多数人与 LLMs 和文档的交互方式是这样的：

```
上传一堆文件 → LLM 在查询时检索相关片段 → 生成答案
```

**问题**：每次提问，LLM 都是从零开始重新发现知识。没有积累。

### LLM Wiki 的解决方案

Karpathy 的类比来自软件工程：**编译**。

```
你不会每次运行程序都重新编译源代码。你编译一次，然后运行二进制文件。

Karpathy 说：用同样的方式对待知识。
你的 PDF 和笔记是源代码。Wiki 是二进制文件。
```

**核心思想**：LLM **增量��建并维护一个持久的 wiki**——一个结构化的、互联的 markdown 文件集合。每次添加新文档时，LLM 不会只是索引它，而是：
- 读取它
- 提取关键信息
- 集成到现有 wiki
- 更新相关页面
- 标记矛盾之处

---

## 为什么这很强大

Karpathy 本人的话：

> "最近我发现非常有用的一件事：使用 LLMs 构建各种主题的个人知识库。想象 Obsidian 是 IDE，LLM 是程序员，wiki 是代码库。你很少（几乎不）自己写 wiki——LLM 写并维护所有内容。你负责 sourcing、探索和提问。LLM 做所有的苦力活——总结、交叉引用、归档和记账。"

**关键区别**：wiki 是一个**持久的、复利的产物**。
- 交叉引用已经在那里了
- 矛盾已经被标记了
- 综合已经反映了你的所有阅读

知识会随着你添加的每个来源和你提出的每个问题而变得更加丰富。

## 架构：三层结构

```
┌─────────────────────────────────────────────────────────┐
│                    Layer 3: Schema                     │
│              (CLAUDE.md / AGENTS.md)                   │
│        告诉 LLM 如何组织 wiki 的配置文档                 │
└─────────────────────────────────────────────────────────┘
                          ▲
                          │ 生成
┌─────────────────────────────────────────────────────────┐
│                     Layer 2: Wiki                       │
│              (LLM 生成和维护)                          │
│    摘要、实体页面、概念页面、比较、综合                 │
└─────────────────────────────────────────────────────────┘
                          ▲
                          │ 读取
┌─────────────────────────────────────────────────────────┐
│                   Layer 1: Raw Sources                  │
│                    (原始来源)                          │
│        文章、论文、图片、数据文件——不可变              │
└─────────────────────────────────────────────────────────┘
```

### Layer 1：Raw Sources（原始来源）

你精心收集的源文档集合。文章、论文、图片、数据文件。**这些是不可变的**——LLM 读取它们但从不修改它们。这是你真理的来源。

### Layer 2：The Wiki（Wiki）

LLM 生成和维护的 markdown 文件目录。摘要、实体页面、概念页面、比较、综合。**LLM 完全拥有这一层**。它创建页面，在新来源到达时更新它们，维护交叉引用，保持一切一致。

> 你读它；LLM 写它。

### Layer 3：The Schema（Schema）

一个文档（如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md），告诉 LLM：
- wiki 如何组织
- 约定是什么
- 在摄入来源、回答问题或维护 wiki 时遵循什么工作流程

这是关键的配置文件——它使 LLM 成为**有纪律的 wiki 维护者**而不是随意的对话伙伴。

## 操作

### 1. Ingest（摄入）

将新的来源放入 raw 集合，然后告诉 LLM 处理它。

**示例流程**：

```
1. LLM 读取来源
2. 与你讨论关键要点
3. 在 wiki 中写摘要页面
4. 更新索引
5. 更新相关实体/概念页面
6. 检查新的矛盾或一致性
```

### 2. Query（查询）

直接向 wiki 提问。

**示例流程**：

```
1. LLM 在 wiki 中查找相关信息
2. 如果需要，读取原始来源
3. 综合回答，并引用相关 wiki 页面
```

### 3. Lint（检查）

定期检查 wiki 健康状态。

**检查内容**：
- 孤立页面（没有链接到任何页面）
- 断链（指向不存在的页面）
- 过时信息（与最新来源矛盾）

## 特殊文件

两个特殊文件帮助 LLM（和你）在 wiki 增长时导航。

### index.md（索引）

面向内容的目录——wiki 中每个页面的列表，带链接和一行描述。

**示例结构**：

```markdown
# Index

## AI
- [[AI/LLM]]
- [[AI/Transformer]]

## Docker
- [[Docker/绑定挂载]]

## 文学
- [[文学/红楼梦]]
```

### log.md（日志）

按时间顺序的记录——记录摄入、问题和变更。

**示例结构**：

```markdown
# Log

## 2026-04-21

### Ingest
- [[.raw/articles/attention-is-all-you-need.pdf]]

### New Pages
- [[AI/Attention]]

### Questions
- Transformer 和 RNN 的区别？
```

---

## 文件命名约定

- 使用小写字母
- 用连字符分隔：`machine-learning.md`
- 中文用中文或拼音：`机器学习.md` 或 `ji-qi-xue-xi.md`
- 避免嵌套超过一层

---

## 页面格式模板

```markdown
# 页面标题

## 概述
（一句话说明这是什么）

## 核心内容
（详细笔记）

## 相关概念
- [[相关页面1]]
- [[相关页面2]]

## 来源
- [[.raw/...]]
```

## 适用场景

### 研究
- 研究论文、博客文章、YouTube 视频、播客
- 建立理解该领域概念的 wiki

### 专业
- 会议记录、客户电话、项目文档
- 让新团队成员浏览整理好的 wiki，而不是挖掘 Slack 历史

### 个人
- 追踪自己的目标、健康、心理学、自我提升
- 记录日记条目、文章、思维

---

## 为什么这有效

维护知识库中最繁琐的部分不是阅读或思考——而是**记账**。

LLM 不会厌倦，可以一次处理 15 个文件，保持 wiki 维护近零成本。
