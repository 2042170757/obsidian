# 给大模型开卷考：RAG 从原理到落地

> [!abstract] 概述
> RAG（检索增强生成）是让大模型在回答问题前先"翻书查资料"的技术。本文从为什么需要 RAG 出发，拆解完整工作流程、六大切片策略、Rerank 机制、常见问题排查，以及 2024-2025 年的最新实践（Agentic RAG、Graph RAG、混合检索等）。

---

## 一、为什么我们需要 RAG？

传统大语言模型（LLM）存在三大根本性短板：

| 短板 | 说明 | 举例 |
|------|------|------|
| **知识过时** | 模型训练完成后，无法获取截止日期之后的新信息 | ChatGPT 初始版本的知识截止于 2021 年，对之后的新法规、新技术完全无知 |
| **幻觉问题** | LLM 基于概率生成文本，会自信地编造听起来合理但实际错误的内容 | 模型可能基于不存在的法规断言企业合规性 |
| **私有数据无法访问** | 通用模型训练于公开数据，无法触及企业内部专有信息 | 模型基于通用退货政策回答 30 天，而实际知识库中电子产品退货是 15 天 |

> [!warning] 重要认知
> RAG 只能**大幅缓解**幻觉，不能**完全消除**。即使在 RAG 加持下，主流模型仍有约 17-25% 的幻觉率。更危险的是：如果检索到的上下文不充分或不相关，反而可能让模型**更自信地给出错误答案**（Google Research, ICLR 2025）。RAG 不是幻觉的"解药"，而是"抑制器"。

**来源**：Lewis et al., 2020 (NeurIPS)；PrivacyBrain 论文 (arXiv 2410.02825) 实验表明加上语义 RAG 后性能提升 40%。

---

## 二、什么是 RAG？

**RAG（Retrieval-Augmented Generation，检索增强生成）** = 检索 + 增强 + 生成。

> [!tip] 核心比喻：**开卷考试 vs 闭卷考试**
> - **传统 LLM** = **闭卷考试**：仅靠"脑子里的记忆"（训练时学到的参数知识）回答问题
> - **RAG** = **开卷考试**：可以先翻阅"参考书"（外部知识库），找到相关资料后再作答

基础流程只有三步：

```
用户提问 → 【检索】从知识库找相关文档 → 【增强】把文档注入提示词 → 【生成】LLM 基于参考资料作答
```

> [!info] 不止一种 RAG
> "开卷考"是最简版的 **Naive RAG**。2024-2025 年学界已识别出 5 种 RAG 范式：
> 1. **Naive RAG** — 一次检索 + 一次生成
> 2. **Advanced RAG** — 增加查询重写、Rerank、上下文压缩等优化环节
> 3. **Modular RAG** — 可插拔的功能模块（记忆模块、路由模块等）
> 4. **Graph RAG** — 用知识图谱增强检索
> 5. **Agentic RAG** — LLM 自主决策多轮检索、自我反思、工具调用

**原始论文**：Lewis et al., "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks", NeurIPS 2020 (Meta AI)。提出 RAG-Sequence 和 RAG-Token 两个版本，在 Natural Questions 上达到 44.5 EM，超越 T5-11B 的 34.5 EM。

## 三、RAG vs 微调：如何选择？

> [!important] 关键认知转变
> 大纲原表述"RAG 是首选"过于绝对。更准确的表述是：**RAG 是推荐的起点**，约 70% 的场景适合从 RAG 开始。但在某些特定场景下，微调更优。2026 年业界最佳实践已转向 **RAG + 微调混合方案**。

### 核心区别

- **RAG 解决"知识问题"**：模型不知道某个信息 → 从外部知识库补充
- **微调解决"行为问题"**：模型的回答风格、格式、推理方式不对 → 改变模型权重

### 多维度对比

| 维度 | RAG | 微调 |
|------|-----|------|
| **前期成本** | 低（$500-$5,000） | 高（$5,000-$20,000+） |
| **知识时效性** | ⭐ 实时更新（改知识库即可） | ❌ 冻结在训练时刻 |
| **可解释性** | ⭐ 可追溯来源 | ❌ 黑盒权重 |
| **技术门槛** | 较低（2-4 周上线） | 较高（4-12 周，需数据集经验） |
| **推理延迟** | 较慢（200-1000ms+，需检索） | ⭐ 快（50-200ms） |
| **输出一致性** | 受检索质量波动 | ⭐ 行为内化，稳定 |
| **大批量成本** | 每次查询有 Token 开销 | ⭐ 训练成本前置，推理便宜 |
| **离线部署** | 需外部知识库 | ⭐ 可完全离线运行 |

### 微调更优的场景

- 推理延迟要求 <200ms（实时对话）
- 输出格式需要严格一致性（法律合同、医疗报告）
- 离线/气隙环境部署
- 高查询量 + 知识稳定不常变化（>1 亿查询/月）

### 2026 最佳实践：混合方案

**RAFT**（UC Berkeley / Microsoft / Meta）：微调模型使其学会**何时信任检索结果、何时依赖内部知识**。混合方案在完整性得分上表现最优（4.50 vs 纯 RAG 4.20 vs 纯微调 3.30）。Harvey AI（法律 AI 独角兽）就是微调 + RAG 的典型生产案例。

## 四、元数据：不只是"文件属性"

> [!note] 大纲原比喻
> "元数据就像 Windows 文件的属性——创建时间、大小"——这个比喻正确但不完整。

在 RAG 中，元数据是**检索过滤和结果验证的关键基础设施**，远不止静态文件属性：

| 元数据类型 | 在 RAG 中的作用 |
|-----------|----------------|
| 文档来源/作者 | 来源追溯、权威性加权 |
| 时间戳 | 知识时效性判断 |
| 切片位置（页码/段落） | 定位原文 |
| 权限标签 | 访问控制 |
| 自定义标签 | 分类过滤、主题检索 |

> [!tip] 关键原则
> "如果你的 RAG 系统没有使用任何元数据，那你可能做错了什么。"（deepset, 2024）
>
> arXiv 2025 年论文证实：元数据前缀嵌入可将检索准确率从基线一致提升，其中公司名称和年份是最强的消歧义信号。

## 五、RAG 的完整工作流程

RAG 系统分为两大阶段。把它们理解为"图书馆建馆"和"读者借书"：

> [!tip] 生活化比喻
> **离线索引 = 图书馆建馆**：买书 → 分类编目 → 贴标签 → 上书架。这个过程可以慢慢做，做一次用很久。
>
> **在线推理 = 读者借书**：读者提问 → 图书管理员查目录 → 找到相关书籍 → 摘录关键段落 → 交给读者。这个过程必须快（毫秒级）。

### 阶段一：离线索引（建图书馆）

```
原始文档 → 数据加载 → 文本切片 → 向量化 → 向量存储
                ↓            ↓          ↓          ↓
          PDF/网页/数据库   拆成小块   转成数字   存入向量数据库
```

### 阶段二：在线推理（借书答疑）

```
用户提问 → 查询重写 → 向量化查询 → 混合检索 → Rerank → 上下文组装 → LLM 生成 → 返回+引用
```

> [!warning] 容易被忽略的环节
> - **查询重写**：用户问"怎么退款"和"退货流程"可能指向同一份文档，需要把口语化问题改写成检索友好的形式
> - **Rerank**：向量检索召回 20 条，Rerank 精筛到 4-6 条再送入 LLM（详见第七章）
> - **语义缓存**：两个语义相似的问题（"如何重置密码" vs "忘记密码怎么重置"）应命中相同缓存——这是生产系统的最大成本节省器

## 六、文本切片策略

文本切片是 RAG 效果的关键影响因素之一。同文档、同嵌入模型，不同切片策略可能导致检索准确率翻倍或减半。

> [!tip] 切片的三大意义
> 1. **适配模型上下文**：LLM 和嵌入模型都有上下文窗口限制
> 2. **提升检索精度**：小块更容易被精确匹配，减少噪音
> 3. **控制成本**：嵌入和检索的 Token 成本与切片大小直接相关

---

### 6.1 固定长度字符切片

**核心原理**：按预设字符数（如每 1000 个字符）均匀切分。

| 维度 | 说明 |
|------|------|
| 优势 | 实现极简单、速度极快、确定性高 |
| 局限 | 容易在句子中间或概念中途生硬截断，破坏语义 |
| 适用 | 快速原型验证、非生产系统、基线测试 |

> [!warning] 中英文差异
> 中文 1 个汉字 ≈ 1.5-2 个 Token，同样的 1000 字符在中文中包含更多语义内容。纯字符切片不考虑 Tokenizer 差异，可能导致 Token 超限。

---

### 6.2 固定长度 Token 切片

**核心原理**：按固定 Token 数（如每 512 Token）切分，精准控制模型输入长度。

| 维度 | 说明 |
|------|------|
| 优势 | 精准控制 Token 用量，避免超出模型上下文 |
| 局限 | 与字符切片一样存在语义破坏风险 |
| 适用 | 对 Token 用量敏感的场景、简单文本 |

---

### 6.3 递归字符切片（LangChain 标配）

**核心原理**：从大到小分层切分——先按段落(`\n\n`)，太长则降为句子(`。`)，再降为分句(`；`)，以此类推。

| 维度 | 说明 |
|------|------|
| 优势 | 尽量不在句子中间切断，保留段落/句子结构 |
| 局限 | 对非标准格式文本（代码、表格、无标点文本）效果差 |
| 适用 | **通用场景首选**——效果与效率的最佳平衡点 |

> [!tip] 注意事项
> 递归切片保证不在句子中间切断（structural completeness），但**不保证每个切片是语义完整的单元**（semantic coherence）。真正的语义完整性需要语义切片或 Agentic Chunking。

---

### 6.4 语义切片 🔬

**核心原理**：计算句子间的语义相似度，当相似度低于阈值时判定发生"主题漂移"，在此处切分。

```
句子1 → 句子2（相似度 0.9 ✓ 同主题）
句子2 → 句子3（相似度 0.3 ✗ 主题变了！→ 在此切分）
```

| 维度 | 说明 |
|------|------|
| 优势 | 语义无损，每个切片是连贯的语义单元 |
| 局限 | 计算成本高（需逐句计算嵌入相似度）；阈值调参困难 |
| 适用 | 内容逻辑复杂的长文本、专业知识问答 |

> [!info] 不只一种实现方式
> 除相似度阈值法外，还可以用 **K-means 聚类**按语义将句子分组，或用 LLM 辅助判断主题变化。后者就是下面要讲的 Agentic Chunking。

---

### 6.5 父子文档切片 🏗️

**核心原理**：双重存储与分离机制——**子切片用于检索，父文档用于上下文**。

```
原文档（父文档，完整保留上下文）
   ├── 子切片 1（200 Token，用于向量检索匹配）
   ├── 子切片 2（200 Token，用于向量检索匹配）
   └── 子切片 3（200 Token，用于向量检索匹配）

检索时：用子切片 2 命中用户问题
生成时：把子切片 2 所属的完整父文档送入 LLM
```

> [!tip] 生活化比喻
> 就像**图书馆的索引卡片 + 原书**系统。索引卡（子切片）很小、搜索快，找到目标后，管理员不是只给你看卡片，而是把**整本原书**（父文档）递给你。

| 维度 | 说明 |
|------|------|
| 优势 | 兼顾细粒度检索的召回率和大篇幅上下文的理解能力 |
| 局限 | 成本与复杂度提升（双倍存储、双倍管理） |
| 适用 | 长文本问答、需要完整上下文的场景（法律合同、学术论文） |

---

### 6.6 Agentic Chunking（智能体切片）🤖

> [!warning] 分类说明
> 大纲将此类称为"大模型切片"。根据学术界共识，更名为 **Agentic Chunking（智能体切片）**。它使用 LLM 自主决策切分，属于语义切片的增强版——两者都是自适应粒度，但决策机制不同：语义切片用嵌入相似度阈值，Agentic Chunking 用 LLM 的语义理解能力。

**核心工作流程**：

```
Step 1: 命题提取（Propositionization）
    "李华，他昨天去了北京" → ["人物：李华", "李华昨天去了北京"]

Step 2: 动态分组（Dynamic Grouping）
    LLM 判断每个命题应归入哪个已有分组，或创建新分组

Step 3: 标题/摘要生成
    为每个分组生成描述性标题和摘要，作为元数据附加到切片中
```

| 维度 | 说明 |
|------|------|
| 优势 | 语义理解最精准、自动解析代词、动态适应内容结构变化 |
| 局限 | 成本高（每次切片调用 LLM API）、速度慢、结果不稳定 |
| 适用 | 高精度需求 + 预算充足、专业领域（医疗、法律、科研） |

> [!info] 新技术：Meta-Chunking (2024.10)
> Meta-Chunking 框架证明：小模型通过困惑度计算和边缘采样策略也能实现高质量切分，不需要强指令遵循能力的大模型。这降低了 Agentic Chunking 的成本门槛。

## 七、切片策略选型指南

```
                    ┌──────────────────┐
                    │   你的场景是？    │
                    └───────┬──────────┘
                            │
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
   快速原型           通用生产场景         高精度需求
        │                   │                   │
        ▼                   ▼                   ▼
  固定字符切片        递归字符切片         语义切片 或
  （最简单）         （最佳平衡点）      Agentic Chunking
                                            │
                                    需要完整上下文？
                                        │
                                    ┌───┴───┐
                                    ▼       ▼
                                   是      否
                                    │       │
                                    ▼       ▼
                             父子文档切片  语义切片
```

**一句话选型**：

| 场景 | 推荐策略 | 理由 |
|------|---------|------|
| 快速原型 | 固定字符切片 | 实现最简单 |
| **通用场景（推荐）** | **递归字符切片** | 效果与效率的最佳平衡 |
| 对 Token 敏感 | 固定 Token 切片 | 精准控制用量 |
| 高精度 + 复杂长文本 | 语义切片 | 语义无损 |
| 长文本 + 需要完整上下文 | 父子文档切片 | 双层索引，兼顾精度与完整 |
| 超高精度 + 预算充足 | Agentic Chunking | 极致智能 |

## 八、Rerank 重排序

> [!tip] 生活化比喻
> **Rerank 就像招聘的"二面"**。HR 先海筛 100 份简历（向量召回），选出 20 份（Top-K），然后用人部门经理（Rerank 模型）精读重排，最终只推荐 5 人进入终面（送入 LLM）。

**核心逻辑**：

```
向量召回（粗筛）              Rerank（精筛）
      ↓                           ↓
Bi-encoder（双塔）           Cross-encoder（交叉）
速度快，精度较低             速度慢，精度极高
召回 Top-20~50              精筛至 Top-3~6
```

**完整链路**：

```
文档切片 → 向量化入库 → 向量召回 (Top-20) → Rerank 重排序 (Top-5) → 送入大模型
```

**为什么需要 Rerank？**
向量相似度 ≠ 语义相关性。向量检索基于整体语义的粗略近似，容易被"看起来像但实际不相关"的文档骗过去。Rerank 逐对计算用户问题和每个文档的精确匹配度，类似"逐字对读"。

**常用 Rerank 模型**：

| 模型 | 特点 |
|------|------|
| Cohere Rerank | 商业 API，效果最好，开箱即用 |
| bge-reranker-v2 | 开源，中文友好，BAAI 出品 |
| Jina Reranker | 开源，多语言支持好 |
| mxbai-rerank | 开源，轻量快速 |

## 九、重叠度（Overlap）

> [!tip] 生活化比喻
> 就像**用剪刀剪一条绳子编成手环**——如果在每段绳子的端点处不留一点重叠，手环的连接处就会断裂。重叠度就是相邻切块之间故意重复的那一小段文字。

**作用**：防止把一句话、一段逻辑硬生生切断。当用户在两句交界处提问时，相关内容至少会在其中一个切片中完整出现。

**最佳实践**：

| 参数 | 推荐值 |
|------|--------|
| **通用起步** | 10-20% 重叠（如 512 Token 切片 + 50-100 Token 重叠） |
| 大块切片（>1024 Token） | 10% 重叠 |
| 标准切片（512 Token） | 15% 重叠 |
| 小块切片（256 Token） | 20-25% 重叠 |

> [!warning] 调整顺序
> 重叠度不能替代糟糕的切片策略。调试顺序：**先调 chunk_size 和分隔符，再调重叠度**。

**来源**：NVIDIA 2024 基准测试证实 15% 重叠在 FinanceBench 数据集上表现最优。

## 十、RAG 常见问题排查

### 问题一：知识库有答案，但模型回答不符合

**排查链路**（从后往前查）：

```
原文档有无知识片段？
    ↓ 有
切片后有无这个片段？
    ↓ 有
检索结果有无知识片段？（检查 top_k）
    ↓ 有
召回的有无知识片段？
    ↓ 有
提示词是否要求模型"按知识库回答"？
    ↓ 是 → 问题可能在提示词设计或模型"固执"程度
```

常见陷阱：提示词说"参考以下资料回答"，但没说"**只**参考以下资料回答"。模型可能混合了参数知识和检索知识。

### 问题二：模型"忘记"前面说了什么

> [!warning] 技术事实
> LLM 没有"记忆"机制——它不是在"储存"和"遗忘"对话。每次推理都是根据当前上下文窗口进行纯计算。"忘记"实际上是**对话历史超出上下文窗口后被截断**，模型不再能"看到"早期的对话内容。学界将这个问题称为 **"Lost in the Middle"** 或 **Context Rot**（上下文衰减）。

**解决方案**：

| 情况 | 方案 |
|------|------|
| 还没超出上下文窗口 | 加强提示词，把关键信息放在对话开头和结尾（模型对中间位置注意力最弱） |
| 快超出上下文窗口 | 压缩/摘要早期对话 |
| 上下文已被污染 | 重开一个新对话 |

### 问题三：为什么模型运行一段时间后爆内存？

**根因**（按可能性排序）：
1. **KV Cache 未释放** — 每次推理产生的键值缓存越积越多（最主要原因）
2. **历史对话未清理** — 对话历史持续累积
3. **历史向量未释放** — RAG 检索结果向量持续占用显存
4. **模型被重复加载** — 错误地多次加载模型实例

> [!tip] 症状时间线
> 刚启动 → 干净 → 能跑 → 用几次 → 垃圾越积越多 → 显存/内存占满 → 爆了

### 问题四：综合诊断 —— 8 种常见失败模式

| 故障模式 | 症状 | 根因 |
|----------|------|------|
| 切片太碎 | 检索到的片段缺少上下文 | chunk_size 过小 |
| 切片太大 | 检索结果噪音多 | chunk_size 过大 |
| 嵌入不匹配 | 检索不相关 | 嵌入模型与数据语言/领域不匹配 |
| 查询漂移 | 口语问题搜不到正式文档 | 缺少查询重写 |
| 幻觉增强 | RAG 后反而更错 | 检索到不相关/错误上下文（Google 2025 发现） |
| 引用缺失 | 无法验证答案 | 提示词未要求模型标注引用 |
| 权限泄露 | 用户看到了不该看的内容 | 缺少元数据权限过滤 |
| 语义缓存误命中 | 相似但不同的问题返回相同答案 | 缓存相似度阈值设置过高 |

## 十一、如何评估 RAG 效果

> [!info] 核心原则
> "你无法优化你无法衡量的东西。"——这是 RAG 领域的共识格言。

### RAGAS 评估框架

RAGAS（Retrieval Augmented Generation Assessment）是 RAG 领域最广泛使用的评估框架，提供四大核心指标：

| 指标 | 衡量什么 | 为什么重要 |
|------|---------|-----------|
| **Context Precision** | 检索到的文档中有多少是真正相关的 | 检索质量 |
| **Context Recall** | 应该检索到的文档有多少被检索到了 | 检索覆盖度 |
| **Faithfulness** | 生成的回答是否完全基于检索到的上下文 | 幻觉检测 |
| **Answer Relevancy** | 回答是否直接回应了用户问题 | 回答质量 |

2025 年新增 **Context Utilization** 指标，衡量模型实际使用了多少检索到的上下文（检索到了 ≠ 模型看了）。

## 十二、前沿扩展（2024-2025）

> [!note] 进阶阅读
> 以下主题大纲未涉及，但它们是 RAG 从"能用"到"好用"的关键。

| 主题 | 一句话说明 |
|------|-----------|
| **Graph RAG** | 用知识图谱实体和关系增强检索，适合多跳推理 |
| **Agentic RAG** | LLM 自主决策多轮检索、调用工具、自我反思 |
| **多模态 RAG** | 不止检索文本，还能检索图片、表格、图表 |
| **查询重写** | RAG 系统的"翻译官"——把口语问题改写成检索友好形式 |
| **混合检索** | 向量检索 + 关键词检索（BM25），互补长短，生产系统标配 |
| **HyDE** | 让 LLM 先生成一个"假设答案"，用这个答案去检索——有时比你原始问题检索得更准 |

## 比喻词典

> 以下比喻帮助理解 RAG 各环节的核心机制。

| 概念 | 比喻 |
|------|------|
| **向量嵌入** | 把每本书的内容压缩成一串 GPS 坐标——语义相近的书在坐标空间中靠得很近 |
| **语义相似度检索** | 像在地图上找最近的加油站——"最近的"就是语义最相似的 |
| **Rerank** | 招聘二面——HR 海筛 100 份简历（向量召回），部门经理精读重排推 5 人（Rerank） |
| **父子文档切片** | 图书馆索引卡片 + 原书——用卡片快速找到目标，但把整本书递给你 |
| **语义切片** | 像看电视剧时靠片头曲变化来判断"这集结束了"——主题变了就切 |
| **离线索引 vs 在线推理** | 图书馆建馆（慢慢做）vs 读者借书（必须快） |
| **重叠度** | 剪绳子编手环时连接处留的余量——防止在接口处断裂 |
| **上下文窗口** | LLM 的"桌面"——只能看到桌面上摆着的资料，放不下的就看不见了 |

## 核心知识点回顾

1. **RAG 的三大意义**：解决知识过时、缓解幻觉、打通私有数据
2. **RAG 不是幻觉的"解药"**：只能缓解，不能消除；不充分的检索上下文反而会增强幻觉
3. **两大阶段**：离线索引（建图书馆）+ 在线推理（借书答疑）
4. **切片选型**：递归字符切片是通用场景的最佳平衡点
5. **必备增强**：Rerank + 混合检索 + 查询重写 = 生产级 RAG
6. **2026 最佳实践**：RAG + 微调混合方案而非二选一
