# 容器与虚拟机

## 概述
理解容器技术的核心前提：容器解决了什么问题，以及它与传统虚拟机的本质区别。

---

## 一、容器解决了什么问题？

### 1.1 传统部署的痛点："能跑起来" vs "到处乱跑"

在容器出现之前，软件部署面临一个经典难题：

| 场景 | 描述 |
|------|------|
| **开发者说"在我机器上能跑"** | 开发环境和生产环境的 OS、库版本、配置不同，导致代码在生产环境崩溃 |
| **环境配置地狱** | 每台服务器都需要手动安装依赖、配置环境，耗时且容易出错 |
| **资源浪费** | 为了隔离不同应用，每台服务器只跑一个服务，硬件利用率低 |
| **扩展困难** | 新增服务器需要重复整个环境搭建流程，无法快速扩容 |

### 1.2 容器化解决的核心问题

| 问题 | 容器的解决方案 |
|------|----------------|
| **环境不一致** | 应用 + 依赖 + 配置打包成一个镜像，"Build once, run anywhere" |
| **部署复杂** | 一条命令 `docker run` 启动完整环境，秒级完成 |
| **资源浪费** | 多个容器共享宿主机内核，密度远高于虚拟机 |
| **扩展困难** | 容器秒级启动，配合编排工具（如 K8s）实现自动扩缩容 |
| **隔离性** | 每个容器独立运行，互不干扰（进程级隔离） |

## 二、虚拟机 vs 容器：架构对比

### 2.1 架构图

```
        虚拟机 (VM)                          容器 (Container)
┌─────────────────────────┐        ┌─────────────────────────┐
│     Application A       │        │  App A  │  App B  │ App C│
├─────────────────────────┤        ├─────────┴─────────┴──────┤
│     Guest OS            │        │   Bins/Libs (共享)        │
├─────────────────────────┤        ├──────────────────────────┤
│     Virtual Hardware    │        │   Container Runtime      │
├─────────────────────────┤        │   (containerd / runc)    │
│     Hypervisor          │        ├──────────────────────────┤
├─────────────────────────┤        │   Host OS Kernel (共享)   │
│     Host OS             │        ├──────────────────────────┤
├─────────────────────────┤        │   Host OS                │
│     Physical Hardware   │        ├──────────────────────────┤
└─────────────────────────┘        │   Physical Hardware      │
                                   └──────────────────────────┘
```

**关键区别：**
- **虚拟机**：每个 VM 包含完整的 Guest OS，通过 Hypervisor 虚拟硬件
- **容器**：共享宿主机内核，只打包应用和依赖库，通过 Namespace + Cgroup 实现隔离

### 2.2 隔离机制差异

| 维度 | 虚拟机 | 容器 |
|------|--------|------|
| **隔离级别** | 硬件级（Hypervisor） | 操作系统级（Namespace + Cgroup） |
| **内核** | 每个 VM 独立内核 | 共享宿主机内核 |
| **隔离强度** | 非常强（VM 逃逸难度极高） | 较强（但共享内核是攻击面） |
| **CVE 影响范围** | 仅影响单个 VM | 内核漏洞可能影响所有容器 |

> **安全提示：** 共享内核是容器的根本安全权衡。对于多租户或不可信工作负载，可考虑 gVisor、Kata Containers 或 Firecracker 等沙箱方案。

### 2.3 镜像体积对比

| 指标 | 虚拟机 | 容器 |
|------|--------|------|
| **磁盘占用** | 10-100 GB / 每个 VM | 100 MB - 1 GB / 每个镜像 |
| **原因** | 包含完整 OS + 驱动 + 系统服务 | 只包含应用 + 依赖，分层共享 |

### 2.4 启动效率对比

| 指标 | 虚拟机 | 容器 |
|------|--------|------|
| **启动时间** | 30 秒 - 5 分钟（完整启动） | 100 - 500 毫秒 |
| **原因** | 需要引导内核、初始化 init 系统、启动系统服务 | 直接创建进程，无需引导 OS |

### 2.5 资源开销对比

| 指标 | 虚拟机 | 容器 |
|------|--------|------|
| **内存开销** | 512 MB - 2 GB / 每个 VM（Guest OS + 系统守护进程） | 1 - 5 MB / 每个容器（Namespace + Cgroup 簿记） |
| **CPU 开销** | 5-15%（硬件虚拟化 + Guest OS 调度） | < 1-2%（接近原生性能） |
| **磁盘 I/O** | 原生的 80-95%（虚拟化驱动） | 原生的 95-100% |
| **网络延迟** | 较高（虚拟化网卡） | 接近原生 |

### 2.6 什么时候用什么？

| 场景 | 推荐方案 |
|------|----------|
| 微服务、CI/CD、云原生应用 | **容器** |
| 需要不同内核/OS 的环境 | **虚拟机** |
| 多租户、合规性要求高 | **虚拟机**（或 VM 内跑容器） |
| 快速扩缩容、高密度部署 | **容器** |
| 遗留应用、完整 OS 环境 | **虚拟机** |

> **生产实践：** 大多数现代架构是两者结合——VM 提供基础设施层的强隔离边界，VM 内部运行 Docker 容器实现应用部署的密度和一致性。
