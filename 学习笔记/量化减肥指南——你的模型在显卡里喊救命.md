# 量化减肥指南——你的模型在显卡里喊救命

> 大模型太胖了，显卡扛不住？来给你的模型减减肥！

---

## 概述

模型量化（Quantization）和知识蒸馏（Knowledge Distillation）是两种核心的模型压缩技术，目标相同：让大模型在消费级硬件上跑起来。量化通过降低数值精度减少显存和计算量，蒸馏则通过"师生传承"将大模型的能力迁移到小模型。

> **一句话理解**：量化是把高清照片压缩成 JPEG（体积小、细节略有损失），蒸馏是请名师辅导出一个优秀学生（学生学到精华，但不需要老师的全部知识）。

---

## 为什么我们需要模型压缩？

- 原始大模型的硬件门槛高
- 推理成本与响应速度的瓶颈
- 端侧部署的存储与算力限制

三大主流"瘦身"方案解析：
- 模型量化

## 模块一：模型量化

### 什么是量化？

想象你有一个装满货物的仓库（模型权重），每个货物都用精确到小数点后 6 位的标签标注重量（FP32）。

- **量化前**：每件货物标签写 "3.141593 克"，需要很大的标签纸
- **量化后**：每件货物标签只写 "3 克"，标签纸变小了，仓库也变小了
- **代价**：重量不那么精确了，但对于大多数用途来说 "3 克" 足够了

这就是量化的核心思想：**用更少的位数表示数值，换取更小的模型体积和更快的计算速度**。

### 问题：大模型的显存困境

| 模型规模 | FP32 显存 | FP16 显存 | INT4 量化后 | 压缩比 |
|----------|-----------|-----------|-------------|--------|
| 1B 参数  | 4 GB      | 2 GB      | 0.5-0.7 GB  | ~6x    |
| 7B 参数  | 28 GB     | 14 GB     | 5-7 GB      | ~4x    |
| 13B 参数 | 52 GB     | 26 GB     | 9-12 GB     | ~4x    |
| 70B 参数 | 280 GB    | 140 GB    | 40-50 GB    | ~5x    |

> **注意**：以上显存为模型权重的理论值，实际运行时还需加上 KV Cache、激活值、框架开销等，通常需要额外 30-50% 的显存。

### 核心定义&价值

- 核心思想：映射变换
- 核心目标：降本增效
- 关键前提：精度可控
- 三大价值：存储压缩、速度提升、成本降低

---

### 数值精度的对比

把数值精度想象成尺子的刻度：

| 精度   | 比喻                | 位数     | 每参数字节数   | 7B模型存储  | 精度等级 |
| ---- | ----------------- | ------ | -------- | ------- | ---- |
| FP32 | 游标卡尺（0.01mm 精度）   | 32-bit | 4 Bytes  | ~28 GB  | 最高   |
| FP16 | 普通直尺（1mm 精度）      | 16-bit | 2 Bytes  | ~14 GB  | 较高   |
| BF16 | 工程尺（1mm 精度，但量程更大） | 16-bit | 2 Bytes  | ~14 GB  | 较高   |
| FP8  | 厘米刻度尺             | 8-bit  | 1 Byte   | ~7 GB   | 中    |
| INT8 | 厘米刻度尺             | 8-bit  | 1 Byte   | ~7 GB   | 中等   |
| INT4 | 只有 16 个刻度的尺子      | 4-bit  | 0.5 Byte | ~3.5 GB | 较低   |

#### FP16 vs BF16 核心区别

| 特性    | FP16 (半精度)     | BF16 (Brain Float 16) |
| ----- | -------------- | --------------------- |
| 符号位   | 1 bit          | 1 bit                 |
| 指数位   | 5 bit          | 8 bit                 |
| 尾数位   | 10 bit         | 7 bit                 |
| 数值范围  | ±65,504        | ±3.39×10³⁸（与FP32相同）   |
| 精度    | 较高             | 较低                    |
| 训练稳定性 | 需要Loss Scaling | 更稳定，无需Loss Scaling    |

**BF16在大模型训练中的优势**：
- 避免梯度溢出（动态范围与FP32相同）
- 无需Loss Scaling，省去额外调参
- A100/H100 GPU有专用BF16计算单元
- 减少NaN/Inf问题，收敛性更好

---

### 量化的收益有多大？

| 收益 | FP32→INT8 | FP32→INT4 |
|------|----------|----------|
| **存储压缩** | 4倍压缩 | 8倍压缩 |
| **显存节省** | 28GB→7GB (7B模型) | 28GB→3.5GB (7B模型) |
| **速度提升** | 1.2-2倍（GPU） | 1.5-3倍（GPU） |

> **注意**：速度提升高度依赖硬件支持（如NVIDIA Tensor Core、Intel VNNI/AMX），在支持INT8 Tensor Core的GPU上（如A100），INT8 GEMM可达约2倍加速。INT4的速度提升更依赖具体实现和硬件。

### 量化的技术分类：PTQ-训练后量化

**比喻**：想象你要教学生做菜：

- **PTQ（训练后量化）**：学生已经毕业了，你让他把做菜步骤简化一下。学生不需要重新学做菜，只需要把复杂的步骤用简单的方式表达出来。速度快，但可能有些细节丢失。

**定义**：在模型训练完成后，直接对权重和激活值进行量化，不需要重新训练。

**流程**：
```
1. 加载预训练模型
2. 准备少量校准数据集（几百~几千条）
3. 运行校准数据，收集激活值统计信息
4. 计算量化参数（scale、zero-point）
5. 执行量化转换
6. 验证量化后模型精度
```

**优点**：
- 速度快（分钟级完成）
- 简单易用，工具链成熟
- 资源需求低，不需要GPU长时间训练
- 数据需求少（只需少量校准数据）

**缺点**：
- 精度下降明显（尤其4-bit或更低）
- 对异常值（outlier）敏感
- 缺乏补偿机制，模型无法主动适应量化误差

**典型方法**：GPTQ、AWQ、SmoothQuant、LLM.int8()

---

### 量化的技术分类：QAT-量化感知训练

**比喻**：学生还在学校时，你就告诉他 "考试只能用简化的步骤"，让他从一开始就学习如何在简化条件下做好菜。效果更好，但需要重新训练。

**定义**：在训练过程中模拟量化误差的影响，让模型学会适应低精度计算。

**流程**：
```
1. 准备完整训练数据集
2. 在前向传播中插入伪量化节点（Fake Quantization）
3. 模拟低精度计算过程
4. 反向传播使用直通估计器（STE）近似梯度
5. 模型学习适应量化误差
6. 训练完成后导出量化模型
```

**核心机制**：
- **伪量化节点**：在前向传播中模拟量化-反量化过程
- **直通估计器（STE）**：解决量化函数不可导的问题，近似计算梯度

**优点**：
- 精度更高，模型能学习适应量化误差
- 支持极低比特量化（4-bit、2-bit甚至二值化）
- 权重分布向量化友好方向调整

**缺点**：
- 训练成本高，需要GPU资源和时间
- 实现复杂，需要插入伪量化节点
- 数据需求大，通常需要完整训练数据

**典型方法**：PyTorch FX Graph Mode Quantization、LSQ、BRECQ

---

### PTQ vs QAT 核心区别

| 特性 | PTQ | QAT |
|------|-----|-----|
| **时机** | 训练后 | 训练过程中 |
| **是否需要训练** | ❌ 不需要 | ✅ 需要 |
| **校准数据量** | 少量（几百~几千条） | 较多（完整训练集） |
| **精度损失** | 较大（尤其低比特） | 较小 |
| **速度** | 非常快（分钟级） | 较慢（需训练） |
| **计算资源** | 少 | 多 |
| **适用场景** | 快速部署、精度要求不极端 | 精度要求高、低比特量化 |

**大模型时代的选择**：
- INT8量化且精度要求中等 → PTQ
- INT4/INT2或精度要求极高 → QAT
- 大模型（LLM）→ 主流用PTQ（GPTQ、AWQ、SmoothQuant），因为QAT训练成本太高

### 主流量化算法：RTN

**全称**：Round To Nearest

**核心思想**：最简单的量化方式——四舍五入。

```
原始值: 0.7, 0.3, -0.5, 0.9
INT4 量化: 1, 0, -1, 1  (直接四舍五入到最近的整数)
```

**比喻**：RTN 就像一个粗心的会计，记账时直接把所有数字四舍五入到整数。大部分时候没问题，但如果有一笔特别大的交易（异常值），四舍五入就会造成巨大误差。

**关键局限性**：
- **对异常值极其敏感**：如果某个权重值特别大（比如 100），它会"霸占"大部分量化范围，导致其他正常大小的权重（0.1~1.0）被挤压到少数几个刻度上，精度损失严重
- **实际应用中很少单独使用**：通常作为其他方法的 baseline 对比

```
问题示例：
权重: [0.5, 0.3, -0.2, 100.0]  # 100.0 是异常值

INT4 量化范围: -128 ~ 127
- 100.0 占据了大量刻度
- 0.5, 0.3, -0.2 被压缩到极少数刻度
- 结果：正常值精度损失巨大
```

### 主流量化算法：GPTQ

**全称**：Accurate Post-Training Quantization for Generative Pre-trained Transformers

**论文**：Frantar et al., ICLR 2023

**核心思想**：基于**最优脑损伤（Optimal Brain Damage）框架，逐列量化权重，每量化一列后用 Hessian 信息补偿误差，将量化误差"均匀分散"到尚未量化的列上，从而最小化整体输出误差。

**比喻**：想象一个精密天平工厂要降级生产（量化）。RTN 的做法是把所有零件直接换成粗糙零件（四舍五入），误差累积无人管。GPTQ 的做法则是：
1. **逐个零件**更换——每次只换一列权重
2. **换完就校准**——换完一个零件后，用 Hessian 矩阵（二阶导数信息）计算这个更换对整体精度的影响
3. **补偿误差**——把误差分摊到还没换的零件上，让它们"代偿"
4. **最终结果**——所有零件都降级了，但整体精度损失被最小化

这就是 GPTQ 的精髓：**量化不是孤立的四舍五入，而是一个有误差补偿的有序过程**。

**数学原理**：

```
目标：最小化量化后的输出误差

  min ||W·X - Q(W)·X||²

核心公式（逐列量化）：

  对于第 j 列权重 w_j：
  1. 量化: w_j → quant(w_j)   （四舍五入到最近量化值）
  2. 计算误差: δ_j = w_j - quant(w_j)
  3. 补偿未量化列: W[:, j+1:] -= δ_j · H^{-1}_{jj} · H[:, j+1:]

其中:
- X 是校准数据（128 条左右即可）
- H = 2·X·X^T 是 Hessian 矩阵（二阶信息）
- H^{-1} 通过 Cholesky 分解高效求解
- 补偿的本质：用二阶梯度信息指导误差的最优分配
```

**关键流程**：

```
输入：预训练模型 + 少量校准数据（128条）

1. 计算 Hessian 矩阵 H = 2·X·X^T（一次计算，全局复用）
2. Cholesky 分解求 H^{-1}
3. 逐列遍历权重矩阵：
   a. 将当前列量化到目标精度（如 INT4）
   b. 计算量化误差 δ
   c. 利用 H^{-1} 将误差最优地分摊到剩余列
4. 重复步骤3，直到所有列都量化完成
5. 输出量化后的模型
```

**为什么 Hessian 信息如此重要？**

```
类比：考试成绩调整

没有 Hessian（RTN）：
  - 每道题直接扣分，不管题目重要性
  - 简单题和大题一视同仁
  - 最终成绩波动大

有 Hessian（GPTQ）：
  - 知道每道题对总分的"影响权重"
  - 量化误差优先从不重要的列补偿
  - 最终成绩（输出精度）损失最小
```

**关键技术细节**：

| 特性 | 说明 |
|------|------|
| 目标精度 | W4A16（4-bit 权重，16-bit 激活）为主 |
| 校准数据 | 仅需 ~128 条样本（速度极快） |
| 量化速度 | 几分钟内完成（7B 模型约 3-5 分钟） |
| 分组量化（Group Quantization） | 每 128 个权重共享一组 scale/zero-point，平衡精度与效率 |
| Cholesky 分解 | 保证数值稳定性，避免 Hessian 求逆的精度问题 |
| 逐列处理 | 保证每步补偿的最优性（贪心但理论可证最优） |

**优势**：
- 精度损失小：4-bit 量化后精度下降通常 < 1%
- 速度快：几分钟完成，无需重训练
- 生态成熟：AutoGPTQ、Transformers 原生支持
- 支持 vLLM/TGI 等推理框架的高效部署

**局限性**：
- 依赖校准数据的选择（不同数据分布可能影响量化质量）
- 对激活值不做处理（只量化权重），激活中的异常值问题需要其他方法解决
- 量化速度比 AWQ 略慢（Hessian 计算开销）
- 逐列处理无法天然并行化

**与 RTN 的对比**：

| 维度 | RTN | GPTQ |
|------|-----|------|
| 量化策略 | 直接四舍五入 | 逐列量化 + 误差补偿 |
| 误差处理 | 无 | Hessian 指导的最优补偿 |
| 校准数据 | 不需要 | 需要 ~128 条 |
| INT4 精度 | 差（异常值敏感） | 好（误差被分散） |
| 计算开销 | 最低 | 中等（Hessian 分解） |

### 主流量化算法：SmoothQuant

**核心思想**：通过数学等价变换，将激活值中难以量化的异常值转移到权重上，使激活值变得"平滑"。

**比喻**：SmoothQuant 像一个聪明的税务顾问。他发现：
- 激活值（收入）有异常高的部分，直接量化会"交很多税"（精度损失大）
- 权重（资产）分布比较均匀

他的策略：把一部分"收入"转移到"资产"中，让两者的分布都变得均匀，这样量化时的"税率"（精度损失）就降低了。

**数学原理**：
```
原始: Y = X · W  (X 有异常值)

平滑后: Y = (X · diag(s)^{-1}) · (diag(s) · W)
        = X' · W'

其中 s_j = max(|X_j|)^α / max(|W_j|)^{1-α}

α ∈ [0, 1] 控制转移程度：
- α = 0：不转移（保持原样）
- α = 1：全部转移
- α = 0.5：平衡转移（通常效果最好）
```

**关键特点**：
- 目标精度：W8A8（权重和激活都用8bit）
- 解决了激活值难以量化的问题
- 避免混合精度带来的额外开销
- 精度损失小

**优势**：
- 实现了权重和激活的统一8bit量化
- 推理速度提升显著
- 精度保持良好

### AWQ（4bit）

**核心思想**：并非所有权重同等重要，与显著激活通道（salient channels）对应的权重对模型精度影响更大。

**比喻**：想象一个乐队在录音。AWQ 发现：
- 小提琴（重要权重）的声音特别突出，需要保留更多细节
- 三角铁（不重要权重）的声音很小，可以压缩多一些

AWQ 不是直接保护小提琴，而是把三角铁的音量调大一点（等效缩放），这样所有乐器在量化时都能获得更公平的对待。

**为什么只保护 1% 的权重就能提升精度？**

因为这 1% 的权重对应的是"异常激活通道"——它们的激活值比其他通道大 10-100 倍。这些通道虽然数量少，但对输出的贡献巨大。保护它们，就是保护模型的"关键少数"。

**等效变换公式**：
```
原始: y = Wx
缩放后: y = (W · diag(s)) · (diag(s)^{-1} · x)

其中 s 是缩放因子，对重要通道 s > 1（放大），
量化后再乘回去，等效于保护了重要权重的精度
```

**关键创新**：
- 只保护1%的显著权重，就能显著提升精度
- 通过缩放而非混合精度来保护重要权重
- 不需要重训练

**优势**：
- 精度损失小（相比GPTQ更好）
- 推理速度快
- 支持W4A16（4bit权重，16bit激活）

### GGUF - 通用的CPU/GPU兼容格式（重点）

**全称**：GPT-Generated Unified Format（由 llama.cpp 项目推广）

**比喻**：GGUF 格式像一个智能收纳盒：
- **K-quant**：不是所有东西都用同样大小的盒子，而是根据重要性选择不同大小的盒子
- **M/S/L 后缀**：代表收纳盒的 "紧凑程度"
  - **M (Medium)**：平衡方案，盒子大小适中
  - **S (Small)**：更紧凑，盒子更小，但可能有些东西放不下（精度损失）
  - **L (Large)**：更宽松，盒子更大，东西放得更精确

**起源**：
- 最初为llama.cpp项目设计
- 逐步成为CPU推理的标准格式
- 被Ollama、LM Studio、koboldcpp等广泛采用

**跨平台兼容性**：
- 文件内包含完整的模型元数据（tokenizer、架构信息等）
- 自包含格式，单文件即可加载
- 支持CPU (AVX/ARM NEON)、GPU (CUDA/Metal/Vulkan)等多后端
- 在多种工具中通用

### 常见 GGUF 量化类型

| 类型 | 位数 | 说明 | 适用场景 |
|------|------|------|----------|
| Q4_0 | 4 位 | 基础 4 位量化 | 快速推理 |
| Q4_K_M | 4 位 K-quant | 4 位混合精度（Medium） | 平衡选择 |
| Q4_K_S | 4 位 K-quant | 4 位混合精度（Small） | 极致压缩 |
| Q5_K_M | 5 位 K-quant | 5 位混合精度（Medium） | 高精度需求 |
| Q6_K | 6 位 K-quant | 6 位量化 | 接近 FP16 精度 |
| Q8_0 | 8 位 | 基础 8 位量化 | 最高量化精度 |

**K-quant 的核心思想**：
- 对模型的第一层和最后一层使用更高精度（因为它们对输出影响最大）
- 对中间层使用较低精度
- 不同层使用不同的量化位数，实现"混合精度量化"

### 量化的 Trade-off

量化的本质是**用精度换资源**，但这个交换并非无脑操作，需要在多个维度之间做权衡。

---

#### 1. 精度 vs 效率的平衡

**核心矛盾**：位数越低 → 模型越小、推理越快 → 但精度损失越大。

```
FP32 ──→ FP16 ──→ INT8 ──→ INT4 ──→ INT2/INT3
 │         │        │        │         │
 └──精度最高        │        │    精度最低
           └──速度适中       └──速度最快
```

**经验法则**：

| 量化级别 | 精度损失 | 适用建议 |
|----------|----------|----------|
| FP16 | 几乎无损 | 精度敏感场景（医疗、金融） |
| INT8 | < 1% | 大多数生产部署的默认选择 |
| INT4 | 1%~3% | 资源受限的推理部署 |
| INT2/INT3 | 3%~10%+ | 仅用于实验或对精度要求极低的场景 |

> **关键判断**：精度损失是否"可接受"取决于下游任务。聊天机器人对 1% 的精度波动不敏感，但代码生成或数学推理可能因此出现明显错误。

---

#### 2. 精度 vs 性能的博弈

量化不仅影响"答对率"，还影响推理延迟和吞吐量：

| 维度 | 量化后提升 | 代价 |
|------|-----------|------|
| **显存占用** | 大幅下降（INT4 可降 4~8 倍） | 无明显代价 |
| **推理延迟** | 通常降低（内存带宽瓶颈缓解） | 需要硬件支持对应低精度指令 |
| **吞吐量** | 显存省了 → 可以同时跑更多请求 | 量化/反量化引入额外计算开销 |
| **首次加载时间** | 模型文件更小，加载更快 | 量化过程本身需要额外时间 |

**反直觉的情况**：
- 量化并不总是更快——如果 GPU 不支持 INT4/INT8 Tensor Core，反而可能因为量化/反量化的额外开销**变慢**
- 在 compute-bound 场景下（如超长序列的 prefill），量化带来的加速有限，瓶颈在计算而非内存带宽

---

#### 3. 硬件环境的限制

不同硬件对量化的支持程度差异巨大：

| 硬件 | 支持的精度 | 说明 |
|------|-----------|------|
| **NVIDIA A100/H100** | FP16/BF16/INT8/FP8 | Tensor Core 原生支持，INT8 加速明显 |
| **NVIDIA RTX 30/40 系列** | FP16/INT8 | INT4 需要 bitsandbytes 等软件库 |
| **Apple Silicon (M1-M4)** | FP16/INT8 | ANE（神经引擎）支持 INT8，GGUF 表现优异 |
| **纯 CPU** | INT8/INT4（软件模拟） | 没有硬件低精度单元，加速主要靠内存节省 |
| **边缘设备（手机/NPU）** | INT8/INT4 | 通常有专用 NPU，INT4 支持因芯片而异 |

**选型建议**：
- 有 NVIDIA GPU → 优先用 GPTQ/AWQ（GPU 加速明显）
- Apple Silicon 或纯 CPU → 用 GGUF（llama.cpp 优化良好）
- 边端设备 → 检查 NPU 支持的最低精度，通常 INT8 是安全线

---

#### 4. 业务场景的适配

没有"最好的量化"，只有"最合适的量化"：

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| **聊天/对话** | INT4（AWQ/GPTQ） | 用户对微小精度损失不敏感，省显存优先 |
| **代码生成** | INT8 或 INT4+校准 | 代码对精度要求更高，错误会编译失败 |
| **数学推理** | FP16 或 INT8 | 数值计算对精度敏感 |
| **RAG 检索增强** | INT4 | 检索结果已提供上下文，模型精度要求相对低 |
| **多租户服务** | INT4 + vLLM | 显存省了 → 同一张卡能服务更多用户 |
| **端侧离线** | GGUF Q4_K_M | 纯 CPU 推理，跨平台兼容 |
| **科研/微调** | FP16/BF16 | 量化会影响梯度，训练阶段保持高精度 |

---

#### 总结：量化的决策框架

```
精度要求高？
├── 是 → INT8（最安全）或 FP16（无损）
└── 否 → 硬件有 GPU？
         ├── 是 → INT4（AWQ 优先，GPTQ 其次）
         └── 否 → GGUF Q4_K_M / Q5_K_M
```

> **一句话总结**：量化不是"越低越好"，而是"在你的硬件、业务和精度要求之间找到甜蜜点"。

## 模块二：知识蒸馏

### 什么是知识蒸馏？

**比喻**：知识蒸馏就像一个经验丰富的老师（大模型）教学生（小模型）：

- **老师**：看到题目后，不仅给出正确答案，还会说 "我觉得 A 的可能性 60%，B 的可能性 30%，C 的可能性 10%"
- **学生**：不仅要学会老师给的答案，还要学会老师这种"思考方式"
- **软标签**：老师给出的概率分布（60%, 30%, 10%）比硬标签（A 是对的）包含更多信息

**师生框架**：
- **教师模型（Teacher）**：大型、复杂、预训练好的模型，具有较高准确率
- **学生模型（Student）**：较小、轻量的模型，目标是学习教师模型的"知识"

---

### 硬标签与软标签

| 类型 | 定义 | 示例 | 信息量 |
|------|------|------|--------|
| **硬标签** | 传统one-hot编码标签 | [0, 0, 1, 0] | 仅"是/否"二元信息 |
| **软标签** | 教师模型输出的概率分布 | [0.05, 0.10, 0.80, 0.05] | 包含类别间相似性信息 |

**软标签的关键优势**：
- 包含"暗知识"（Dark Knowledge）
- 例如：猫的图片，软标签显示"猫(0.80)、虎(0.10)"，说明猫和虎有相似之处
- 小模型直接从数据学不到的这些关系，通过教师的软标签可以学到

---

### 温度系数的秘密

**比喻**：温度系数 T 就像调节老师的 "耐心程度"：

- **T = 1**：老师很直接，只说 "A 是对的"（标准 softmax）
- **T = 2**：老师更耐心，会说 "A 的可能性比较大，B 也有点可能"（分布更平滑）
- **T = 5**：老师非常耐心，会把所有可能性都说出来（分布非常平滑）

**数学公式**：
```
软标签: q_i = exp(z_i / T) / Σ_j exp(z_j / T)

其中:
- z_i 是模型输出的 logits
- T 是温度系数（T ≥ 1）
- T 越大，分布越平滑，"暗知识" 越明显
```

**T 的选择**：
- T = 1：标准 softmax，硬标签
- T = 2~5：常用范围，能暴露类别间的关系
- T > 10：分布过于平滑，信息量反而减少

---

### 蒸馏损失函数

**蒸馏损失 = 硬标签损失 + 软标签损失**

```
L = α · L_hard(y, p) + (1-α) · T² · L_soft(q_T, p_T)

其中:
- L_hard：学生预测与真实标签的交叉熵
- L_soft：学生软标签与老师软标签的 KL 散度
- α：平衡系数（通常 0.1~0.5）
- T²：缩放因子，补偿温度 T 对梯度的缩放效应

注意：T² 缩放因子是必要的，因为当 T > 1 时，
软标签的梯度会被 1/T² 缩小，需要乘以 T² 来补偿
```

---

### 蒸馏的技术分类

#### 离线蒸馏

**定义**：教师模型预先训练完成，再将知识迁移给学生模型。

**流程**：
```
1. 先训好Teacher
2. 冻结Teacher
3. 训练Student
```

**优点**：简单易实现，Teacher和Student训练完全解耦
**缺点**：依赖高质量预训练Teacher，Teacher能力上限决定了Student的上限
**LLM应用**：DeepSeek-R1-Distill系列、Alpaca、Vicuna

#### 在线蒸馏

**定义**：教师模型和学生模型同时训练，相互学习。

**流程**：
```
1. Teacher和Student同步更新参数
2. 相互学习，共同提升
```

**优点**：Teacher也在持续学习，避免Teacher能力固定化
**缺点**：训练复杂度高，计算资源需求大
**LLM应用**：多模型协作训练、推测解码（Speculative Decoding）

#### 自蒸馏

**定义**：模型自身既充当Teacher又充当Student。

**流程**：
```
1. 用模型深层/后期的输出指导浅层/早期的输出
2. 或用前一轮的输出指导当前轮
```

**优点**：不需要额外的Teacher模型，成本低
**缺点**：蒸馏效果可能不如真正的Teacher-Student架构
**LLM应用**：Self-Instruct、STaR（Self-Taught Reasoner）

---

### 三种蒸馏方式对比

| 维度 | 离线蒸馏 | 在线蒸馏 | 自蒸馏 |
|------|---------|---------|--------|
| Teacher是否预训练 | ✅ 是 | ❌ 同步训练 | 自身即是Teacher |
| 训练复杂度 | 低 | 高 | 中 |
| 计算成本 | 中 | 高 | 低 |
| 知识质量 | 稳定 | 动态提升 | 有瓶颈 |
| 适用场景 | 模型压缩/部署 | 持续学习 | 无外部Teacher时 |

---

### 量化和蒸馏的区别？

- **量化**：无损结构 - 保持模型架构不变，只降低数值精度
- **蒸馏**：重塑模型 - 用小模型学习大模型的知识，架构可以完全不同
- **结合**：极致优化 - 可以先蒸馏再量化，实现最大压缩比

---

### 蒸馏的应用场景

1. **端侧部署**：将大模型能力迁移到手机、嵌入式设备
2. **高并发推理**：小模型推理速度快，适合高并发场景
3. **领域适配**：针对特定领域进行蒸馏，提升专业能力

## 模块三：如何通过模型名称判断量化信息

### 常见量化后缀含义

| 后缀 | 全称 | 含义 | 特点 |
|------|------|------|------|
| **GPTQ** | Accurate Post-Training Quantization for Generative Pre-trained Transformers | 训练后量化方法 | GPU推理优化，CUDA加速 |
| **AWQ** | Activation-aware Weight Quantization | 激活感知权重量化 | 精度损失更小，推理更快 |
| **GGUF** | GPT-Generated Unified Format | 统一格式 | CPU推理优化，跨平台 |

### 三种格式对比

| 特性 | GPTQ | AWQ | GGUF |
|------|------|-----|------|
| 主要运行设备 | GPU | GPU | CPU (也可GPU) |
| 生态系统 | Transformers/vLLM | Transformers/vLLM | llama.cpp/Ollama |
| 文件格式 | .safetensors | .safetensors | .gguf |
| 量化精度 | 较好 | 更优 | 良好 |
| 推理速度 | 快(GPU) | 更快(GPU) | 适合CPU场景 |

### 如何通过模型名称判断量化信息

**示例**：`TheBloke/Llama-2-7B-GPTQ` 或 `TheBloke/Llama-2-7B-AWQ`

- **TheBloke**：上传者/组织名
- **Llama-2-7B**：基础模型名称
- **GPTQ/AWQ**：量化方法

**GGUF命名示例**：`llama-2-7b.Q4_K_M.gguf`

- **llama-2-7b**：基础模型
- **Q4_K_M**：4-bit K-quant Medium量化
- **.gguf**：文件格式

### 选择建议

- **有NVIDIA GPU** → 优先选AWQ（精度和速度最优）或GPTQ
- **CPU/Apple Silicon/低资源** → 选GGUF（配合llama.cpp/Ollama）
- **Hugging Face下载**：GPTQ和AWQ直接用transformers加载，GGUF需要用llama.cpp或Ollama加载

## 模块四：模型参数与硬件的对应关系

### 显存需求的核心公式

**理论公式**：
```
显存 (GB) = 参数量 (B) × 每参数字节数

FP32: 1B 参数 × 4 字节 = 4 GB
FP16: 1B 参数 × 2 字节 = 2 GB
INT8: 1B 参数 × 1 字节 = 1 GB
INT4: 1B 参数 × 0.5 字节 = 0.5 GB
```

**为什么实际显存比理论值大？**

比喻：理论值就像你计算 "搬家公司需要多少辆货车"，只算了货物本身的体积。但实际上还需要：
1. **KV Cache**：推理时存储历史信息的缓存（约占 20-30%）
2. **激活值**：计算过程中的中间结果（约占 10-20%）
3. **框架开销**：PyTorch、CUDA 等框架本身的内存占用（约占 5-10%）
4. **碎片化**：内存分配不连续导致的浪费（约占 5%）

**实际经验公式**：
```
实际显存 ≈ 理论显存 × 1.3 ~ 1.5

例如：7B INT4 理论 3.5 GB，实际约 5-7 GB
```

### 示例：7B INT4 模型

1. **参数显存需求**：70亿 × 0.5字节 = 3.5G
2. **实际总显存**：3.5G × 1.3~1.5 = 5-7G
3. **推荐最低显卡**：8GB显存（如RTX 4060、RTX 3060）

### 常见模型显存参考

| 模型 | INT4显存需求（推理） | 推荐显卡 |
|------|---------------------|---------|
| 7B | ~5-7 GB | RTX 4060 (8GB) |
| 13B | ~10-14 GB | RTX 4070 (12GB) |
| 33B/34B | ~20-24 GB | RTX 4090 (24GB) |
| 70B | ~40-48 GB | A100 (80GB) 或双卡 |

## 速度提升参考

量化后的速度提升取决于多个因素：
- **硬件支持**：NVIDIA GPU 的 INT8 Tensor Core 加速明显
- **内存带宽**：量化减少内存访问，对 memory-bound 任务提升大
- **计算密度**：compute-bound 任务提升相对较小

**典型速度提升范围**（硬件依赖）：
| 量化方式 | 相对 FP16 速度 | 说明 |
|----------|---------------|------|
| INT8 | 1.2x ~ 2x | 需要硬件 INT8 支持 |
| INT4 | 1.5x ~ 3x | 需要专门的 INT4 内核（如 bitsandbytes） |
| 混合精度 | 1.1x ~ 1.5x | 取决于量化比例 |

> **注意**：以上数据为典型范围，实际性能取决于具体硬件（GPU 型号）、框架版本、模型架构等因素。

## 技术选型指南

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| GPU推理（精度优先） | AWQ 4-bit | 精度损失最小，速度最快 |
| GPU推理（兼容性） | GPTQ 4-bit | 生态成熟，工具支持广 |
| CPU/本地部署 | GGUF Q4_K_M | 跨平台，资源需求低 |
| 训练阶段 | BF16混合精度 | 稳定性好，显存节省 |
| 极致压缩 | INT4 + 知识蒸馏 | 精度和大小的最佳平衡 |

## 关键数据速查

- **7B模型 FP32→INT4**：28GB→3.5GB（8倍压缩）
- **7B模型 INT4推理**：实际需要5-7GB显存
- **BF16 vs FP16**：BF16训练更稳定，FP16精度更高
- **AWQ vs GPTQ**：AWQ精度更好，GPTQ生态更成熟
