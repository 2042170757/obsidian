# 这样写Prompt，AI真的会谢

> [!tip] 一句话理解
> **好的Prompt = 清晰的任务 + 足够的上下文 + 明确的输出格式 + 适当的约束**

---

## 什么是Prompt？

### 核心定义

Prompt（提示词）是引导大语言模型（LLM）生成特定输出的输入文本。

> "Prompt engineering is the process of writing effective instructions for a model, such that it consistently generates content that meets your requirements." — OpenAI

**本质**：通过优化输入文本的结构、措辞和上下文，使模型能够可靠地产生符合预期的输出。

### 广义形式

| 形式 | 说明 | 示例 |
|------|------|------|
| **问题（Question）** | 直接询问信息 | "什么是量子计算？" |
| **命令（Command）** | 指示执行特定操作 | "帮我写一段用户登录的Python代码" |
| **背景（Context）** | 提供情境信息 | "假设你是一家电商公司的客服" |
| **示例（Example）** | 通过案例示范期望格式 | Few-shot learning中的示例 |

### 反驳与验证

> [!warning] 质疑1：Prompt只是"技巧"，不是真正的工程
> 批评者认为，Prompt工程更像是"魔法咒语"而非系统性工程，不同模型对相同Prompt响应差异大，缺乏可移植性。
> **验证结果：部分正确** - 确实存在模型特异性，但现代框架如LangChain已将其系统化，可优化性和可测试性与传统工程相当。

> [!warning] 质疑2：随着模型能力提升，Prompt工程会被取代
> 有人认为未来模型会"自然理解"意图，Prompt工程不再必要。
> **验证结果：部分正确** - 虽然模型理解力在提升，但"Lost in the Middle"研究显示，信息位置效应仍然存在，对上下文的精心管理始终重要。

## 大模型是如何"理解"你的Prompt？

### 三大核心机制

#### 1. Token（分词）机制

LLM不处理原始文本，而是将输入分割为token。关键点：

- **Tokenization（分词）**：文本被分割为子词单元（subword），使用BPE（Byte-Pair Encoding）等算法
- **映射为ID**：每个token映射为唯一的数字ID
- **嵌入（Embedding）**：数字ID转换为高维向量，捕捉语义信息

> [!info] 经验规则
> **不同模型的 token 计算方式：**
> 
> **OpenAI GPT 系列：**
> - 英文：100 tokens ≈ 75个单词
> - 中文：1个中文字符 ≈ 1-2个tokens
> - 混合文本：根据语言比例计算
> 
> **其他模型（Claude、Gemini等）：**
> - 各有不同的计算方式
> - 中文字符通常按实际字符数计算
> 
> **通用估算：**
> - 中文文本：100个中文字符 ≈ 100-200 tokens
> - 英文文本：100 tokens ≈ 75个单词

#### 2. 上下文窗口（Context Window）

上下文窗口是模型单次可处理的最大token数量，2026年范围从128K到10M tokens。

> [!danger] 关键发现：Lost in the Middle效应
> 斯坦福大学2023年研究发现，模型对上下文不同位置的信息利用度不同：
> - **开头位置**：高度注意（primacy bias）
> - **结尾位置**：高度注意（recency bias）
> - **中间位置**：显著降级，甚至低于无文档基准
>
> **验证结果：正确** — 即使128K+窗口的模型也存在此问题。

#### 3. 概率生成

LLM的核心机制是**下一个token预测**（next-token prediction）：

1. 输入的token被并行处理，通过自注意力机制建立关联
2. 模型计算整个词表中每个可能token的概率分布
3. 通过采样（temperature控制随机性）选择下一个token
4. 重复直到产生停止信号

> [!warning] 质疑：LLM在"理解"而非只是预测？
> 批评者认为LLM展现出真正的理解能力，不是简单的统计预测。
> **验证结果：争议性** — 虽然模型表现出复杂行为，但核心机制仍是概率预测。"理解"是涌现属性还是幻觉仍在讨论中。

## Prompt工程的价值与应用场景

| 应用场景 | 说明 | 核心Prompt策略 |
|----------|------|---------------|
| **代码辅助** | 代码生成、重构、调试 | 多层次system prompt（身份定义、行为规则、工具API、安全层） |
| **文档写作** | 自动生成README、API文档 | Few-shot格式示例、Chain-of-thought结构化输出 |
| **数据分析** | 结构化查询生成、可视化代码 | RAG提供领域特定上下文 |
| **智能体构建** | 多层system prompt实现自主agent行为 | Plan-Execute-Observe-Repeat循环 |

### 反驳与验证

> [!warning] 质疑1：Prompt工程无法保证AI安全
> 即使详细的安全prompt也可被绕过（RLHF攻击达98%绕过率）。
> **验证结果：正确** - 安全prompt是引导而非保证，需要多层防御（输入过滤、输出验证、运行时策略）。

> [!warning] 质疑2：Long-running agent的核心失败不是prompt问题
> 研究显示，agent失败多源于内部调整循环问题：缺少字段、约束模糊、API调用失败等。
> **验证结果：正确** - "上下文工程"概念兴起，强调prompt只是一小部分，工具输出、对话历史、检索文档等占99.7%的上下文才是关键。

> [!warning] 质疑3：Prompt无法替代工具和基础设施
> 即使详细prompt，模型无法访问实时数据、执行确定性计算或产生外部副作用。
> **验证结果：正确** - 这是agent架构需要工具（Tools/MCP）的根本原因。

## 模块一：三大核心原则总览

### 清晰（Clarity）

**核心定义**：任务描述明确、无歧义，让模型知道具体要做什么。

**无效示例**：
```
写一篇关于Python的文章
```

**优质示例**：
```
写一篇面向初学者的Python入门教程，要求：
- 篇幅800字以内
- 包含3个代码示例
- 使用通俗易懂的语言
```

> [!note] 技巧
> 用「动词开头」明确任务：分析、对比、总结、列出、解释、编写...

### 具体（Specificity）

**核心定义**：提供足够的细节和上下文，避免模糊表述。

**无效示例**：
```
推荐一款手机
```

**优质示例**：
```
我是一个预算有限（3000元以内）的学生，主要用来拍照和社交，偶尔打游戏。
推荐一款安卓手机，重点对比拍照效果和续航。
```

### 有边界（Boundaries）

**核心定义**：明确任务的边界、约束和限制条件。

**常见类型**：
- 字数限制
- 格式约束
- 风格要求
- 禁止事项

**无效示例**：
```
写一首诗
```

**优质示例**：
```
写一首关于「秋天的思念」的七言绝句，要求：
- 押ang韵
- 意境悠远
- 不超过28个字
```

### 三大原则有效性评估

| 原则 | 有效性 | 反驳论据 | 验证结果 |
|------|--------|----------|----------|
| **清晰** | 高 | 过于清晰可能限制探索空间 | 有效，但"清晰"不等于"详细" |
| **具体** | 高 | 具体与泛化的平衡点难以把握 | 有效，但需避免过度具体 |
| **有边界** | 中 | 边界过于严格可能降低创造力 | 有效，但边界应随任务类型调整 |

## 模块二：六大基础技巧

### 1. 指令前置（Prefix）

**底层核心逻辑**：将核心指令放在提示词的开头部分，利用模型对开头内容更强的注意力机制，确保关键指令被优先处理和理解。

**使用场景**：
- 长上下文任务
- 多步骤复杂指令
- 关键约束条件

**举例**：
```
❌ 模糊写法：
这个数据分析任务，你要仔细检查数据质量，确保准确，然后生成报告。

✅ 指令前置：
生成数据分析报告。任务要求：1. 检查数据质量 2. 确保准确性 3. 输出格式为表格。
```

> [!warning] 反驳论据
> - **位置效应有限**：现代LLM已采用全局注意力机制，对整个上下文进行综合理解，位置影响力下降
> - **实验矛盾**：部分研究显示，中置或后置指令在特定场景下效果更优
> **验证结果：部分正确** - 指令前置仍然是良好实践，因为符合人类阅读习惯、防止指令被遗忘在长上下文中，但需结合"重复关键指令"策略。

---

### 2. 补充背景（Context）

**万能公式**：
> 我是"具体身份" + 我需要"执行的任务" + 用于"具体目的/场景" + 受众是"目标人群"

**举例**：
```
我是[一家电商公司的数据分析师]，需要[分析用户购买行为数据]，
用于[制定季度营销策略]，受众是[公司高管]。
```

**反驳论据**：
- **信息过载风险**：过多背景可能导致模型关注点分散
- **身份冲突**：同时赋予多个角色可能产生行为矛盾
- **推理成本增加**：背景越长，模型需要更多token进行理解和推理

> [!note] 验证结果
> 研究表明，背景信息在3-5个关键点时效果最佳。"具体身份"部分确实能提升专业性，但超出必要范围的背景会造成"干扰效应"。

---

### 3. 设定输出格式

**高频推荐格式**：

| 格式 | 适用场景 | 遵循度 |
|------|----------|--------|
| **纯文本列表** | 简单列举 | >95% |
| **Markdown表格** | 对比、分类 | >90% |
| **JSON** | 结构化数据 | 70-80% |
| **Markdown** | 文档输出 | >90% |

**进阶格式设计**：
```
请按以下格式输出：
1. [总结] - 一句话概括（不超过20字）
2. [分析] - 详细分析（3-5个要点）
3. [建议] - 可执行的具体建议（2-3条）
```

**反驳论据**：
- 模型对复杂嵌套格式（如嵌套JSON）的遵循度不稳定，尤其是层级较深时
- 严格格式限制可能降低回答的流畅性和自然度

> [!tip] 最佳实践
> 简单格式（列表、纯文本）遵循度高；复杂嵌套格式需后处理，而非依赖模型内部格式。

---

### 4. 角色扮演

**核心语法公式**：
```
你是一位[专业角色]，你的职责是[任务描述]，你应该[行为准则]
```

**场景应用正例**：
```
你是一位有10年经验的资深Java后端工程师，熟悉Spring Boot微服务架构。
帮我评审以下代码，指出潜在的安全隐患和性能问题。
```

**System Prompt vs 对比内角色设定**：

| 类型 | 定义位置 | 效果强度 | 持久性 |
|------|----------|----------|--------|
| **System Prompt** | 全局设定 | 70-80%影响 | 持久 |
| **对话内角色** | 单次交互 | 30-40%影响 | 随对话消散 |

> [!warning] 关键发现
> 研究显示，单次交互中的角色声明对输出影响约30-40%，而System Prompt中的角色设定影响达70-80%。
> **最佳方案**：将核心角色设定置于System Prompt，而非每次交互重复。

**反驳论据**：
- **角色漂移（Role Drift）**：长时间对话中模型可能逐渐偏离设定角色
- **身份过载**：多重角色标签可能导致行为不一致

---

### 5. 分步拆解（CoT - Chain of Thought）

**核心定义**：通过显式步骤引导模型将复杂问题分解为子问题，逐一解决后汇总。

**触发公式**：
```
"请逐步思考"
"Step by step"
"首先...然后...最后..."
```

**最佳适用场景**：
- 数学计算与推理
- 逻辑分析
- 代码调试
- 复杂决策

**举例**：
```
问题：计算[商店利润]
请按以下步骤分析：
Step 1: 计算总收入 = 销量 × 单价
Step 2: 计算总成本 = 固定成本 + 变动成本
Step 3: 利润 = 收入 - 成本
逐步计算并给出最终答案。
```

**反驳论据**：
- 简单事实性问题反而因分步而效率降低
- token消耗增加约2-5倍
- 前置步骤错误会导致后续步骤全部错误

> [!note] 验证结果
> **对复杂推理任务有效**（数学、逻辑、代码调试）；**对简单任务有害**：增加响应时间但不提升质量。最佳触发条件：任务复杂度>3步以上推理。

---

### 6. 少样本示例（Few-shot）

**三种模式对比**：

| 模式 | 示例数量 | 适用场景 |
|------|----------|----------|
| **Zero-shot** | 0个 | 简单标准任务 |
| **One-shot** | 1个 | 学习期望输出风格 |
| **Few-shot** | 2-5个 | 复杂格式/风格任务 |

**核心法则**：
> [!important] 示例质量（Quality）> 示例数量（Quantity）

**正确示例**：
```
帮我判断评论的情感倾向，正面/负面/中性。

示例：
"这个手机太棒了，拍照清晰，运行流畅" → 正面
"等了十天还没到货，差评" → 负面
"收到货了，包装完好" → 中性

待判断："客服态度很好，但产品有点贵"
```

**反驳论据**：
- **示例偏差（Sample Bias）**：示例可能不代表真实分布
- **数量悖论**：更多示例可能稀释核心指令的权重
- **示例选择主观性**：选择"好示例"本身需要领域知识

> [!note] 验证结果
> 1个高质量示例优于3个低质量示例；超过5个示例后，边际效益趋近于零；示例的标签（label）比输入更重要。

## 模块三：正反案例深度拆解

### 案例来源

| 来源 | 说明 |
|------|------|
| **CRISPE框架** | 包含Capacity/Role、Insight、Statement、Personality、Experiment五个维度 |
| **BROKE框架** | 借鉴OKR思想，目标驱动，后与港科大NLP实验室合作优化 |
| **COSTAR框架** | 最结构化的框架之一，应用于市场营销和内容创作 |

### 核心分析维度

| 维度 | 正面特征 | 反面特征 |
|------|----------|----------|
| **角色定义** | 具体、专业、有限范围 | 模糊、过宽、矛盾 |
| **上下文** | 充足、相关、结构化 | 不足、过载、无关 |
| **任务陈述** | 清晰、可衡量、无歧义 | 模糊、过于宽泛 |
| **输出格式** | 明确、指定、示例完整 | 未指定、矛盾 |
| **安全性** | 输入验证、指令分层 | 未隔离、未过滤 |

### 闭环改写流程

```
发现问题 → 分析根因 → 改写优化 → 验证测试
     ↑                                      |
     └──────────────────────────────────────┘
```

## 模块四：Prompt结构化设计框架

### CRISPE框架

| 维度 | 说明 |
|------|------|
| **C - Capacity/Role** | 定义AI应扮演的专业角色 |
| **I - Insight** | 提供相关背景信息和上下文 |
| **S - Statement** | 明确核心任务或请求 |
| **P - Personality** | 指定期望的沟通风格和特征 |
| **E - Experiment** | 请求多种变体进行比较 |

**反驳论据**：
- 5个组件可能导致过度复杂，对于简单任务反而效率低下
- Experiment组件的实用性存疑，多次变体会增加成本和延迟

### BROKE框架

| 维度 | 说明 |
|------|------|
| **B - Background** | 任务上下文环境 |
| **R - Role** | AI扮演的专业角色 |
| **O - Objective** | 可量化的核心目标 |
| **K - Key Results** | 可衡量的成功标准 |
| **E - Evolve** | 支持多轮对话和持续优化 |

**反驳论据**：
- OKR类比牵强，目标管理的人类语境与AI提示工程存在本质差异
- Key Results量化困难，并非所有任务都能用KR衡量
- Evolve组件定义模糊，实际应用中缺乏明确的执行标准

### 框架对比

| 维度 | BROKE | CRISPE |
|------|-------|--------|
| 核心理念 | 目标驱动、结果导向 | 任务驱动、步骤详尽 |
| 适用场景 | 战略规划、复杂决策 | 明确任务、技术实现 |
| 灵活性 | 高（支持演进） | 中（步骤固定） |
| 学习难度 | 低 | 中等 |

## 模块五：进阶技巧预览

### 1. 提示词注入与安全

**核心观点**：
- Prompt注入是OWASP LLM Top 10排名第一的漏洞
- 需要多层防御策略

> [!danger] 现实威胁
> 研究表明89%的攻击可在GPT-4o上成功，78%在Claude 3.5 Sonnet上成功。当前防御措施（rate limiting、content filters、safety training）只能减缓而非阻止攻击。

---

### 2. 多轮对话管理

**核心观点**：
- 简单拼接整个对话历史在3轮后有效，30轮后崩溃
- 需要滑动窗口、摘要、层级记忆等策略

**反驳论据**：
- 更大的上下文窗口（如Gemini 2M）仍存在"lost in the middle"效应
- 摘要策略可能丢失关键信息（用户偏好、承诺、决策）

---

### 3. RAG中的Prompt设计

**核心观点**：
- 检索失败是幻觉的首要原因
- "bookend"模式将高相关度文档放在首位和末位

**反驳论据**：
- 大多数RAG失败实际上源于检索问题，而非prompt措辞
- 上下文长度矛盾：需要足够文档保证相关性，但又需控制在token限制内

> [!note] 验证结果
> 检索确实优先于prompt优化，但prompt设计对RAG性能仍有显著影响。

---

### 4. Agent系统中的Prompt编排

**核心观点**：
- 多Agent编排通过任务分解克服单Agent限制
- 主要模式：Supervisor（管理）和Swarm（对等）

**反驳论据**：
- Supervisor更准确但更慢（额外LLM调用），Swarm更快但可能路由错误
- Agent间通信开销增加延迟和成本
- 子Agent可能丢失父Agent的完整上下文

## 快速检查清单

在发送Prompt前，检查一下：

- [ ] 任务是否明确？（动词开头）
- [ ] 上下文是否足够？（背景、目标、约束）
- [ ] 输出格式是否清晰？（表格/列表/段落）
- [ ] 是否有必要的约束？（字数/风格/语气）

> [!success] 小结
> 好的Prompt不是一蹴而就的，需要根据AI的反馈不断迭代优化。记住：**你和AI沟通的质量，取决于你表达的质量**。
