## 主题 1：为什么我们需要 RAG？

### 核心概念

传统大语言模型（LLM）存在三大根本性短板：

1. **知识过时（Knowledge Cutoff）**：模型在某个时间点完成训练后，无法获取训练数据截止日期之后的任何新信息。例如 ChatGPT 的初始版本知识截止于 2021 年，对于此后的新法规、新技术、新事件完全无知。

2. **幻觉问题（Hallucination）**：LLM 基于统计预测生成文本，而非检索已验证的事实。模型会自信地编造听起来合理但实际错误的内容。例如，在处理企业隐私合规问题时，LLM 可能基于不存在的法规断言合规性（来源：arXiv 2410.02825 "PrivacyBrain" 论文）。

3. **私有数据隔离（Enterprise Data Gap）**：通用模型训练于公开数据，无法访问企业内部专有信息 -- 产品目录、合规文件、HR 政策、财务记录等。在企业场景下，这导致回答缺乏针对性。

### 网上资料汇总

- **Lewis et al., 2020**, "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks" (NeurIPS 2020): RAG 论文的原动力正是解决 LLM 无法访问和精确操作知识的根本局限。
- **Ksolves Blog (2026)**, "How to Reduce LLM Hallucinations in Enterprise Applications": 指出 RAG 可将幻觉率降低 80-90%，并提供完整的引用追溯能力。
- **NAACL 2024 综述论文** (aclanthology.org/2024.naacl-long.219): 系统性地分析了 LLM 的三个知识缺口点 -- 问题理解不足、知识不足、回忆特定信息能力不足。
- **AWS Machine Learning Blog (2025)**, "Reducing hallucinations in LLM agents with a verified semantic cache": 强调在医疗、金融、法律等关键领域，残余幻觉造成的风险（错误信息、法律责任、用户信任丧失）尤为严重。

### 关键数据/案例

- PrivacyBrain 论文实验：持续预训练提升 16%，加上语义 RAG 后提升 40%（相比原始 Llama），指标翻倍。
- 企业场景典型问题：模型基于通用退货政策回答 30 天，而实际知识库中电子产品的退货政策是 15 天。

### 与其他主题的关联

- 引出主题 2（RAG 是什么）作为核心解决方案
- 与主题 3（RAG vs 微调）直接相关 -- 选择 RAG 还是微调取决于要解决的到底是"知识问题"还是"行为问题"
- 主题 10（问题排查）中的"知识污染"现象与此直接相关

## 主题 2：什么是 RAG？

### 核心概念

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种将信息检索与文本生成相结合的架构。核心理念：在 LLM 生成回答之前，先从外部知识库中检索相关文档，然后将检索到的文档作为附加上下文注入到提示词中，最后让 LLM 基于这些"参考资料"生成回答。

**本质比喻："开卷考试"** -- 传统 LLM 是"闭卷考试"（仅靠参数记忆回答），RAG 则是"开卷考试"（可以翻阅参考材料再作答）。

RAG 完整流程：**检索（Retrieve）--> 增强（Augment）--> 生成（Generate）**

### 网上资料汇总

- **Lewis et al., 2020** (NeurIPS 2020, Meta AI / Facebook AI Research): RAG 的原始论文。提出两个版本：
  - **RAG-Sequence**：整个生成序列使用同一个检索到的文档
  - **RAG-Token**：每个生成 token 可以使用不同的检索文档
  - 架构组件：DPR（Dense Passage Retrieval）作为检索器 + BART 作为生成器
  - 联合训练检索器和生成器，无需直接监督该检索哪个文档
  - 在 Natural Questions 上达到 44.5 EM，超过 T5-11B 的 34.5 EM

- **Meta AI Blog**, "Retrieval Augmented Generation: Streamlining the creation of intelligent natural language processing models": 官方博客解释了 RAG 的设计理念 -- 融合"闭卷"（参数记忆）和"开卷"（非参数记忆）的优势。

- **RAG 的开源实现**已在 HuggingFace Transformers 库中提供，使用仅需 5 行代码。

### 关键数据/案例

- RAG-Sequence 在 Natural Questions: 44.5 EM vs T5-11B: 34.5 EM vs DPR: 41.5 EM
- RAG 即使检索到不包含答案的文档时仍能正确回答（NQ 上 11.8% 准确率），展现生成灵活性
- 可学习的检索器对性能至关重要：阻止梯度传播到检索器会显著降低性能

### 与其他主题的关联

- 主题 5（完整工作流程）是 RAG 架构的工程化拆解
- 主题 6（文本切片）影响 RAG 的第一步 -- 检索质量
- 主题 8（Rerank）是 RAG 检索后的关键优化步骤

## 主题 3：RAG vs 微调：为什么 RAG 是首选？

### 核心概念

RAG 和微调解决**本质不同**的问题：
- **RAG** 解决"模型不知道什么"（知识问题）-- 通过外部知识库补充信息
- **微调** 解决"模型的行为方式不对"（风格/格式/推理问题）-- 通过改变模型权重

**四维度对比：**

| 维度 | RAG | 微调 |
|------|-----|------|
| **成本效益** | 低前期成本（无需训练），但每次查询有检索开销（更多 Token） | 高前期成本（数据准备+GPU 计算），但每次推理成本低 |
| **知识时效性** | 实时更新（只需更新知识库文档） | 冻结在训练时刻，需重新训练 |
| **可解释性** | 高（自然可追溯来源） | 低（黑盒权重） |
| **技术门槛** | 2-4 周上线，较低 | 4-12 周上线，需数据集准备和经验 |

### 网上资料汇总

- **Alex Bobes (2026)**, "RAG vs Fine-Tuning: When to Use Each (With Real Cost Data)": 提供详细的成本对比 -- RAG 前期 $500-$5,000，微调 $50-$20,000+；RAG 幻觉率可从 15-20% 降至 3% 以下。
- **Chanl Blog (2026)**, "Fine-tuning vs RAG: why most teams pick wrong": 提出选择决策树 -- 70% 的实际情况首选 RAG，因为主要是"模型不知道某信息"。
- **AI Bytes (2026)**, "RAG vs Fine-Tuning: 7 Factors That Actually Matter": 详细的七个因素对比表（配置时间、数据新鲜度、前期成本、查询成本、幻觉风险、定制深度、维护负担）。
- **Spaceo.ai (2026)**, "RAG vs Fine Tuning: Which Approach to Choose": 强调 RAG 每次查询消耗 2-5 倍的 Token，在高查询量下需权衡。
- **B EYE (2025)**, "RAG vs. Fine Tuning": 指出 RAG 查询延迟比微调高 30-50%。

### 关键数据/案例

- RAG with GPT-4o: setup $500-$5,000, 月运营 $2,000-$5,000 (10K 查询/天), 每查询 $0.007-$0.017
- 微调 GPT-4o: setup $5,000-$20,000, 月运营 $800-$2,000, 每查询 $0.003-$0.007
- 微调开源模型 (LoRA): setup $50-$300, 月运营 $200-$800, 每查询 $0.001-$0.003
- RAG 幻觉率：从 15-20% 降至 3% 以下
- 混合方案在完整性得分上表现最优（4.50 vs RAG 4.20 vs 微调 3.30）

### 与其他主题的关联

- 主题 1（为什么需要 RAG）的两个问题方向（知识 vs 行为）直接对应这个选择
- 主题 8（Rerank）可以进一步提升 RAG 质量，缩小与微调的质量差距
- 混合方案（RAG + 微调）是 2026 年生产系统的主流做法

## 主题 4：元数据在 RAG 中的作用

### 核心概念

元数据（Metadata）是"关于数据的数据" -- 类似 Windows 文件属性（创建时间、大小、作者等）。在 RAG 系统中，文档块的元数据（标题、来源、日期、分类标签、语言等）可以作为：

1. **检索过滤器**：在语义搜索之前/之后按元数据字段筛选，确保只检索符合权限、时效性、领域约束的文档
2. **排序加权信号**：按文档权威性、时效性等元数据调整检索分数
3. **LLM 上下文补充**：将元数据注入提示词，帮助 LLM 理解信息来源和可信度
4. **安全/权限控制**：按用户权限级别过滤可检索的文档范围

### 网上资料汇总

- **arXiv 2601.11863 (2025)**, "Utilizing Metadata for Better Retrieval-Augmented Generation": 系统研究元数据集成策略，发现 prefix（前缀式）和 unified embeddings（融合嵌入）效果一致优于纯文本基线。元数据整合通过增加文档内聚性、减少文档间混淆来改善嵌入空间几何结构。
  - 代码和 RAGMATE-10K 数据集已开源: github.com/raquibvt/RAGMate
- **Microsoft Architecture Center (2025)**, "RAG Enrichment Phase": 详细的元数据字段列表 -- 标题、摘要、重述、关键词、标签、实体、可回答的问题、来源、语言等，每种对应不同的工具和技术。
- **deepset Blog (2024)**, "Leveraging Metadata in RAG Customization": 指出元数据在 RAG 的预处理、检索、排名、提示词构建全流程中都有应用价值。"如果你的 RAG 系统没有使用任何元数据，那你可能做错了什么。"
- **Microsoft Tech Community (2025)**, "Boost RAG Performance: Enhance Vector Search with Metadata Filters in Azure AI Search": 演示如何在向量搜索中添加元数据过滤器（如电影发行年份），正确筛选出"2010 年后的电影"。
- **Avichala Blog (2025)**, "Metadata Filtering In RAG": 将元数据过滤视为与模型选择同等重要的设计哲学。指出时效性与权威性的权衡 -- 较新来源可能不如经典参考文献可靠，需要在操作查询和合规查询中编码不同的动态权重。

### 关键数据/案例

- 元数据前缀嵌入将检索准确率从基线一致提升
- 公司名称和年份是最强的消歧义信号
- 结构线索（章节标题、页码）提供中等但有价值的信号

### 与其他主题的关联

- 主题 5（工作流程）中的检索阶段可集成元数据过滤器
- 主题 6（切片策略）中，切片时应保留原文档的结构元数据
- 主题 10（问题排查）中，缺少元数据过滤是检索不相关的重要原因之一

## 主题 5：RAG 的完整工作流程

### 核心概念

RAG 系统分为两大阶段：

**阶段一：离线索引阶段（Offline Indexing）** -- 将海量文档转化为结构化的向量知识库

1. **数据加载（Data Loading）**：从各种来源（PDF、网页、数据库、API、Confluence 等）提取原始文本
2. **文本切片（Chunking）**：将长文档拆分为适合嵌入模型和 LLM 上下文窗口的小块
3. **向量化（Embedding）**：使用嵌入模型将每个文本块转换为密集向量（dense vector），捕获语义信息
4. **向量存储（Indexing / Vector Store）**：将向量和元数据存入向量数据库（Pinecone、Weaviate、Chroma、Qdrant、Milvus、pgvector 等）

**阶段二：在线推理阶段（Online Inference）** -- 针对用户提问进行精准检索 + 生成回答

5. **查询处理（Query Processing）**：可选地对用户问题进行改写、拆解、扩展（Query Transformation）
6. **检索（Retrieval）**：将问题向量化，在向量库中做相似度搜索，召回 Top-K 个最相关文本块
7. **重排序（Reranking）**：使用更精确的交叉编码器模型对初检结果重新打分排序，取 Top-N 送入 LLM
8. **生成（Generation）**：将检索结果和元数据注入提示词，LLM 基于此"参考材料"生成回答

### 网上资料汇总

- **InfoDive Labs (2026)**, "RAG Architecture: Building Retrieval-Augmented Generation Systems": 系统性地拆解了从文档摄取到部署的每一阶段，强调演示系统与生产系统的鸿沟。
- **Kestra (2026)**, "RAG Pipeline Orchestration: A Practical Guide": 强调离线数据流水线和在线查询流水线必须解耦，其延迟要求、容错策略、编排工具完全不同。
- **Paul S Cannon (2025)**, "Production RAG Pipelines: Architecture to ROI Guide": 指出生产级 RAG 需要混合搜索（向量+关键词）、重排序、查询转换等高级技术。
- **Appycodes (2026)**, "Building a Production RAG Pipeline": 给出完整的 8 阶段流水线架构图，以及语义缓存在生产中的关键作用。
- **Rost Glukhov (2026)**, "RAG Tutorial: Architecture, Implementation, and Production Guide": 提供详细的摄取管道和查询管道步骤清单，以及各层的评估指标。

### 关键数据/案例

- 离线索引阶段的延迟容忍：分钟到小时；在线推理阶段：毫秒到秒
- 检索建议取 k=20（宽检索），然后通过 Reranker 精确筛选至 4-6 条送入 LLM
- 语义缓存是生产系统的最大成本节省器 -- 两个语义相似的问题（"如何重置密码" vs "忘记密码重置"）应命中相同的缓存答案

### 与其他主题的关联

- 主题 6（切片策略）是离线索引阶段的核心步骤
- 主题 8（Rerank）是在线推理阶段的关键增强
- 主题 9（Overlap）是切片策略的重要参数

## 主题 6：文本切片策略（6种）

### 核心概念

文本切片是 RAG 效果的核心决定因素之一 -- 同一份文档、同一个嵌入模型，不同的切片策略可能导致检索准确率翻倍或减半（来源：Niceboy.org 2025 实践验证）。

**切片的三大意义：**
1. **适配模型上下文**：LLM 和嵌入模型都有上下文窗口限制，必须将长文拆小
2. **提升检索精度**：小块更容易被精确匹配，减少噪音
3. **减少冗余信息**：小块让 LLM 只看到相关信息，避免无关内容干扰

### 策略详解

#### a) 固定长度字符切片（Fixed-size Character Chunking）

- **核心原理**：按预设字符数（如每 1000 个字符）均匀切分文档
- **优势**：实现极简单、速度极快、确定性强、易于 A/B 测试
- **局限**：容易在句子中间或概念中途生硬截断，破坏语义完整性
- **适用场景**：快速原型验证、非生产系统、基线测试
- **来源参考**：LangChain CharacterTextSplitter 文档；POMA AI RAG Chunking Strategies

#### b) 固定长度 Token 切片（Fixed-size Token Chunking）

- **核心原理**：按固定 Token 数量切分（如每 512 Token），而非字符数，因为 LLM 按 Token 计费和处理
- **优势**：精准控制 Token 用量，确保每个块不超出嵌入模型和 LLM 的 Token 限制
- **局限**：依然无视语义边界，可能在句子、段落的语义完整处截断；需要正确的分词器
- **适用场景**：对 Token 预算敏感的场景，严格 Token 限制的生产环境
- **来源参考**：Weaviate "Fixed-Size (Token) Chunking"；Ailog RAG Chunking Strategies 2025

#### c) 父子文档切片（Parent-Child / Hierarchical Chunking）

- **核心原理**：双重存储与分离机制 -- 用小"子块"进行精确检索，但将对应的"父块"（更大的上下文段落）送入 LLM。典型实现：LlamaIndex 的 `ParentDocumentRetriever` 和 `AutoMergingRetriever`
- **优势**：兼顾检索精度（小块精确匹配）和生成质量（大块提供完整上下文） -- 是 2025 年 Stanford 研究中 F1 最高的策略（0.86）
- **局限**：需要双重索引存储（父子块都要存）、检索逻辑更复杂、存储和计算成本增加
- **适用场景**：长文档 QA、政策文件、法律合同、技术手册等强上下文依赖性场景
- **来源参考**：LlamaIndex Hierarchical Node Parser；ML Journey (2026) "Chunking Strategies for RAG"；Ailog RAG (2025) "Advanced Chunking Strategies"

#### d) 大模型切片（LLM-based Chunking）

- **核心原理**：通过设计精细的提示词引导 LLM 基于语义和逻辑理解能力自主判断切分点。本质是让 AI 阅读文档后决定"哪里该切开"
- **优势**：极致智能 -- 能处理不规则结构、跨段落概念、隐式话题转换
- **局限**：极高的时间和金钱成本（每次切片调用 LLM）、速度和效率瓶颈严重、结果依赖提示词质量
- **适用场景**：极高精度要求的专业文本、预算充足且对延迟不敏感的离线处理场景
- **来源参考**：Niceboy.org (2025) "Advanced RAG: A Deep Dive into Document Chunking Strategies"；Firecrawl Blog (2025)

#### e) 递归字符切片（Recursive Character Text Splitter）

- **核心原理**：分层递归切分 -- 采用"从大到小"策略，先用段落分隔符（`\n\n`）尝试切分；若生成的块仍超出限制，降级用句子分隔符（`. `）；再超则降级用空格等。LangChain 的 `RecursiveCharacterTextSplitter` 是标准实现
- **优势**：保留自然语义结构和文档层次，效果与效率的最佳平衡点
- **局限**：在非标准文本（无清晰段落标记、特殊格式、表格、代码混排等）场景下效果下降
- **适用场景**：90% 的通用 RAG 场景 -- 文章、博客、论文、报告等
- **来源参考**：LangChain 官方文档；Chroma 测试中 400 Token 配置达 85-90% Recall

#### f) 语义切片（Semantic Chunking）

- **核心原理**：利用嵌入模型计算句子间的语义相似度（余弦相似度），当相似度低于设定阈值时判定发生"主题漂移"，在该位置切分。LangChain 的 `SemanticChunker` 和 LlamaIndex 的 `SemanticSplitterNodeParser` 是常见实现
- **优势**：语义无损 -- 确保每个块包含完整的主题思想，最大化保持语义完整性；可提升召回率最高 9%（相比固定大小切片）
- **局限**：需要嵌入每个句子计算相似度，索引成本是固定大小切片的 3-5 倍；块长度不均匀（可能产生过大或过小的块），需要配合最大长度兜底策略
- **适用场景**：内容逻辑复杂的长文本、多主题混合页面、对知识问答有极高要求的专业领域
- **来源参考**：LangChain SemanticChunker 文档；Firecrawl Blog (2025)；ACL 2025 Recursive Semantic Chunking 论文

### 性能基准对比

来源：Stanford 研究 (2025 年 1 月) 和 NVIDIA (2024) 基准测试

| 策略 | Precision | Recall | F1 Score | NVIDIA 准确率 |
|------|-----------|--------|----------|--------------|
| 固定大小 (512 tokens) | 0.65 | 0.58 | 0.61 | 基准 |
| 语义切片 | 0.78 | 0.72 | 0.75 | +9% recall |
| 递归字符切片 | - | - | - | 0.648 (页面级最高) |
| 层级切片 | 0.82 | 0.79 | 0.80 | - |
| 父子文档切片 | 0.88 | 0.85 | 0.86 | - |

## 主题 7：切片策略选型指南

### 核心概念

没有"万能"的切片策略。选型需要综合考虑：文档类型、查询模式、延迟预算、计算资源、精度要求。

### 选型建议（综合多个来源）

**快速原型 / 通用场景 -- 首选：递归字符切片**
- 配置建议：512 Token 块大小，50-100 Token (10-20%) 重叠
- 理由：效果与效率的最佳平衡点，LangChain 默认策略，Chroma 测试中 85-90% Recall
- 工具：`RecursiveCharacterTextSplitter` (LangChain)

**对长度敏感 / 简单文本 -- 可选：固定长度 Token 切片**
- 配置建议：根据嵌入模型 Token 限制设置
- 理由：精准控制 Token 用量，避免超出嵌入模型和 LLM 限制

**高精度需求 / 复杂长文本 -- 首选：语义切片**
- 配置建议：需要调整相似度阈值（不同文档类型可能需要不同阈值）
- 理由：语义无损，多主题混合页面检索效果显著优于固定大小
- 代价：索引成本 3-5 倍

**长文本问答 / 需要完整上下文 -- 首选：父子文档切片**
- 配置建议：子块 128-256 Token（用于检索），父块 512-2048 Token（送入 LLM）
- 理由：独特的双层索引结构，兼顾细粒度检索的召回率和大篇幅上下文的理解能力
- 工具：LlamaIndex `ParentDocumentRetriever` / `AutoMergingRetriever`

**超高精度 / 预算充足 -- 可选：大模型切片**
- 配置建议：需要精细设计提示词，可结合递归切片做兜底
- 理由：极致智能，但成本高
- 适用：高风险的合规/法律场景

**结构化文档 (PDF/HTML/Markdown) -- 推荐：结构感知切片**
- 配置建议：使用 `MarkdownHeaderTextSplitter` 或按页面切分
- 理由：NVIDIA 2024 基准测试中页面级切片准确率最高（0.648），方差最低

### 迭代优化方法

来自多个来源的最佳实践：

1. **先跑通再优化**：用递归切片（512 Token + 10-20% 重叠）建立基线
2. **测量实际表现**：在 50-100 个代表性文档上测试 20-30 个真实查询
3. **诊断失败原因**：检查是语义断裂（换语义切片）、上下文依赖（换父子文档）、还是结构不对齐（换结构感知切片）
4. **A/B 对比**：至少测试 2-3 种策略后再做最终决定
5. **关注高层优化**：在切片合理的基础上，检索层（Rerank、混合搜索）的提升往往大于切片策略变动

- 主题 9（Overlap）影响所有策略的参数选择
- 主题 8（Rerank）可以作为切片不足的补偿手段

## 主题 8：Rerank 重排序

### 核心概念

**Rerank（重排序）的核心工作**：输入"用户问题 + 一段文本片段"，输出一个相关性分数，分数越高越匹配，最后按分数重新排序。

**完整流程**：文档切片 --> 向量化入库 --> 向量召回 (Retrieve) --> Rerank 重排序 --> 送入大模型

**为什么要 Rerank？**

向量检索使用 Bi-Encoder（双编码器）架构 -- 问题和文档分别独立编码为向量，然后通过余弦相似度比较。这种方式速度快但精度有限，因为问题和文档的向量是分开计算的，没有交叉注意力，丢失了词组级别的精细匹配。

Rerank 使用 Cross-Encoder（交叉编码器）架构 -- 将 [问题; 文档] 拼接后一起通过 Transformer，所有 Token 之间都有全注意力交互。例如，"利率"在问题中如何关联到文档中的"7-9% APR"可以被精确建模。

### 网上资料汇总

- **BGE Reranker 官方文档** (bge-model.com): 提供详细的 Bi-Encoder vs Cross-Encoder 对比和 Reranker 使用指南。BGE 系列是最流行的开源 Reranker。
- **Cohere Rerank 官方文档** (docs.cohere.com): 提供 Rerank API 的完整使用指南。最新模型 `rerank-v4.0-pro` 支持多语言和半结构化数据(JSON)。
  - Cohere Rerank 3 在 100+ 种语言上表现良好
- **EngineersOfAI (2026)**, "Reranking": 详细解释为什么 Cross-Encoder 比 Bi-Encoder 精确 10-20%，以及两级检索的设计原理。
- **Local AI Master (2026)**, "Reranking & Cross-Encoders for RAG": 提供完整的 2026 年 Reranker 模型对比表和质量提升数据（+5 到 +15 NDCG@10）。

### 常用 Rerank 模型对比

| 模型 | 参数量 | 许可 | MTEB Rerank 分 | 最适合 |
|------|--------|------|---------------|--------|
| BGE-Reranker-v2-m3 | 568M | MIT | 60.4 | **默认首选**；多语言 |
| BGE-Reranker-v2-Gemma | 9B | Apache 2.0 | 64.5 | 极致质量 |
| BGE-Reranker-v2-MiniCPM | 2.7B | Apache 2.0 | 62.3 | 质量+速度平衡 |
| Jina Reranker v2 | 278M | Apache 2.0 | 56.8 | 延迟敏感 |
| mxbai-rerank-large-v1 | 435M | Apache 2.0 | 59.4 | 英文；快速 |
| Cohere Rerank 3 | 闭源 | API | 62.0 | 托管；100+ 语言 |
| Voyage-rerank-2 | 闭源 | API | 61.5 | 托管替代 |

### 关键数据/案例

- 添加 Reranker 通常能将检索准确率提升 10-25%（相比仅用向量搜索）
- Cross-Encoder 比 Bi-Encoder 精确 10-20%
- 典型配置：retrieve top-100 --> rerank to top-10 --> LLM uses top-5
- Cross-Encoder 延迟：50-500ms（处理 100 个候选）
- 绝不用 Cross-Encoder 做首轮检索（100 万文档 = 100 万次前向传播 = 不可行）

- 主题 5（工作流程）中的在线推理阶段 -- Rerank 是检索后的关键增强步骤
- 主题 6（切片策略） -- Rerank 可补偿切片策略的不足，是更高 ROI 的优化手段
- 主题 10（问题排查）中约 15% 的检索不相关因缺少 Rerank 导致

## 主题 9：重叠度 (Overlap)

### 核心概念

**重叠度（Overlap / Chunk Overlap）** 就是切块时前后两块之间重复的文字。它用来防止把一句话、一段逻辑硬生生切断，从而提升 RAG 问答准确率。

**为什么需要重叠度？**

当文档被切分为固定大小的块时，信息可能恰好位于两个块的边界处。例如句子 "认证令牌 24 小时后过期，必须刷新" 可能被切分为：
- 块 A："认证令牌 24 小时后"
- 块 B："过期，必须刷新"

没有任何一个块包含完整的语义信息，导致检索失败。重叠度通过在相邻块之间重复部分内容来解决这个问题。

### 网上资料汇总

- **The Neural Base**, "chunk_overlap: preventing context loss at boundaries": 系统解释重叠度的作用和代价权衡。指出重叠度不能替代糟糕的切片策略 -- 如果语义单元是 300 字符而你用 chunk_size=200，再多重叠也救不回来。
- **StackViv Blog**, "Chunking Strategies for RAG: Size, Overlap & Best Practices": NVIDIA 测试了 10%、15%、20% 重叠值，15% 在 FinanceBench 数据集上表现最佳。
- **BSWEN (2026)**, "What Chunk Size and Overlap Should You Use": 提供分场景的重叠度推荐表格。

### 最佳实践

**通用推荐：10-20% 的块大小作为重叠**

| 块大小 | 推荐重叠 | 比例 |
|--------|---------|------|
| 128 tokens | 10-20 tokens | 8-15% |
| 256 tokens | 25-50 tokens | 10-20% |
| 512 tokens | 50-100 tokens | 10-20% |
| 1024 tokens | 100-200 tokens | 10-20% |
| 2048 tokens | 200-400 tokens | 10-20% |

**按场景调整：**

| 重叠比例 | 适用场景 |
|---------|---------|
| 10% | 大块（1500+），边界损失最小 |
| 15-20% | **标准推荐**（块大小 500-1000） |
| 25-30% | 小块（250-500），关键上下文；密集技术文本 |

**成本考量：**
- 更多重叠 = 更好的召回率 + 更多存储 + 更慢搜索
- 更少重叠 = 更少存储 + 更快搜索 + 可能遗漏边界上下文
- 在 10M+ Token 文档上，10% 重叠增加 1M+ 嵌入调用和 10% 向量存储
- 生产系统可在切片时使用重叠但在存储前去重，但需维护块 ID 映射

### 关键洞察

重叠度不是万能药。如果语义单元本身很大而块大小太小，重叠无法拯救。**许多团队花数周调重叠度，实际上应该先调 chunk_size 和分隔符**。

## 主题 10：RAG 常见问题排查

### 核心概念

RAG 系统在生产中可能遇到多种故障。系统性排查方法：按流水线逐阶段逆向检查（检索 --> 排名 --> 上下文组装 --> 生成）。

### 问题分类与排查路线

#### 问题一：回答不符合知识库内容

根据大纲的排查路线：

1. **检查提示词**：确认 LLM 收到的是"根据知识库回答"还是"忽略知识库回答"的指令
2. **检查检索结果**：检索返回的文档块是否包含正确答案？
3. **检查切片**：原文档中有知识片段 --> 切片后还在不在？--> 检索结果中有没有（top_k 够不够大）？
4. **检查召回**：向量检索是否成功召回相关文档？

补充诊断（来源：LlamaIndex Failure Mode Checklist）：
- 使用"二分测试法"：手动构造完美上下文给 LLM。如果 LLM 仍回答错误 --> 生成问题。如果正确 --> 上游检索/切片问题。
- 约 29.7% 的故障源于切片不当，31.5% 源于检索，32.7% 源于生成阶段

#### 问题二：模型忘记刚刚说了什么

**原因分析**：
- 多轮对话中上下文窗口管理不当
- LLM 的"中间丢失"效应（Lost in the Middle）：位置嵌入导致模型对中间位置的上下文注意力权重降低
- 当检索结果被放在上下文窗口中间位置时，模型容易"忽略"它们

**解决方案**：
- 上下文未超出限制时：加强提示词，明确要求基于完整对话历史回答
- 上下文将要超出时：压缩历史对话（摘要历史、只保留关键信息）
- 实现滑动窗口上下文管理
- 对超长对话使用会话摘要技术

#### 问题三：内存爆炸（模型刚开始能运行，用着用着爆内存）

**原因分析**：
- 模型推理时显存/内存未正常释放，越积越多
- 常见根因：
  1. 没有清理缓存（GPU cache / KV cache 累积）
  2. 历史对话向量未释放（每次查询生成的嵌入累积在内存中）
  3. 模型被重复加载（每次请求重新加载模型而不卸载旧实例）
  4. 向量数据库连接未正确关闭
  5. 嵌入批量处理时 batch size 过大

**解决方案**：
- 定期清理 GPU/CPU 缓存
- 设置对话历史上限（保留最近 N 轮）
- 使用模型单例模式，避免重复加载
- 释放不再使用的嵌入向量和中间结果
- 使用显存监控工具，设置告警阈值
- 分批处理大文档，控制 batch size

#### 其他常见问题（来自全网来源汇总）

| 症状 | 可能阶段 | 根因 | 修复方案 |
|------|---------|------|---------|
| 回答听起来合理但实际错误 | 生成 | 幻觉或参数记忆覆盖 | 强化引用要求、降低 Temperature、强制执行引用 |
| 回答部分正确但不完整 | 检索/切片 | 答案跨块分布、相关块排名太低 | 增大块+重叠、加 Rerank、增 k |
| 回答"不知道"但信息存在 | 检索 | 查询与文档词汇不匹配 | 混合搜索、查询扩展、HyDE |
| 回答错误实体/主题 | 切片/检索 | 上下文不匹配或语义漂移 | 语义切片、元数据过滤 |
| 知识污染（模型用参数记忆覆盖检索结果）| 生成 | LLM 的强先验信念 + 位置效应的中间丢失 | 反事实探针测试、加强提示词"仅根据提供文档回答" |
| 嵌入维度不匹配 | 索引 | 嵌入模型已更换但未重建索引 | 用新模型重新嵌入全部语料 |
| CUDA Out of Memory | 基础设施 | 模型太大或 batch size 太大 | 减少 batch size、量化模型、换小模型 |
| 上下文溢出 | 检索/上下文组装 | 检索块太多导致超出 Token 限制 | 减少 top_k、加 Rerank、设定最大 Token 限制 |

### 诊断方法论

1. **先检查检索**：打印检索结果，确认是否包含答案
2. **二分法测试**：手动构造完美上下文 --> 分诊上游/下游问题
3. **检查 Token 数**：是否接近上下文窗口限制
4. **检查生产特有**：索引版本、缓存、追踪是否正确
5. **分布式追踪**：使用 OpenTelemetry 传播 trace ID，关联各阶段日志

### 与其他主题的关联

- 所有其他主题的知识都是排查工具 -- 切片、检索、Rerank、嵌入模型选择、提示词设计等
- 系统性排查需要对完整工作流程（主题 5）有深入理解
