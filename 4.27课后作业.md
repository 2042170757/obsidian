# **了解tcp的三次握手，四次挥手**

我觉得Tcp的三次握手可以用两个人对讲机通话来比喻：
三次握手就像建立通话前的确认，a和b各拿一个对讲机，只有确认对方的听筒和话筒都是好的，才能放心说话，
第1次握手时a按下通话键问b你能听到吗(SYN)，
第2次握手是b，听到了a的对话。回复他说我听到了，你能听到我说吗(SYN+ACK)，
第3次对话是a听到b的回答后，再说一次，我能听到，没有问题的，我们开始吧(ACK)

**那么为什么一定要三次，不能是两次呢？**
如果是只有两次的话，比如A说完B回了一句收到，但是B的回复没有传到A那里，A不知道， B却傻傻的以为已经接通了，就开始等待了，所以有第3次接通的话，才能让双方都确定已经连接完成

那么四次挥手的话就像结束打电话一样
通话要结束的时候不能随便一方直接关机，因为对方的话可能还没有说完，
tcp是全双工的，意思就是两边可以同时说，也可以同时听，就是两条独立的数据通道，所以说需要分别的关闭
第1次握手就是A这边说，我这边想说的话已经说完了，准备结束通话完毕。(FIN)
第2次握手是B听到后回答说好的知道你要结束了，但是我这边还有几句话要交代，稍等一下完毕。(ACK)
第3次握手是b把话说完之后跟a说，好我这边也讲完了，可以挂断，相当于b也发出了结束请求(FIN)
第4次握手是a听到后回复收到，那我就挂了(ACK)

**那么为什么是4次呢？**
因为当a说完，b有可能还没有全部说完，所以要先回一个知道了，等自己彻底说完之后再独立的通知a我也说完了。
那么有没有特殊情况是三次呢？
就是当a刚好说完的时候，b也刚好说完，那么第2步和第3步就可以合并，但是最标准且最保险的就是4次

## **dns的工作原理**

你的电脑
  │  (1) 请求 www.example.com
  ▼
递归解析器（如 192.168.1.1 或 8.8.8.8）
  │  (2) 问根服务器：.com 的TLD地址？
  │  ◄── 返回 .com TLD 服务器地址
  │  (3) 问 .com TLD：example.com 的权威DNS？
  │  ◄── 返回 ns1.example.com 地址
  │  (4) 问 ns1.example.com：www.example.com 的IP？
  │  ◄── 返回 93.184.216.34
  ▼
(5) 把 IP 返回给你

注意区分两种“查询”：
**递归查询**：客户端要求解析器必须给出最终结果。
**迭代查询**：解析器依次问根→TLD→权威，服务器每次只给一步的指引，不替它往下查。

## **了解幂等**

一个操作执行一次，和执行多次，产生的最终效果完全一样，不会因为重复执行而产生额外副作用。
举个例子：排除两次按会出现取消的现象，你按10次去5楼，电梯只会停在5楼一次，和你按1次效果一样。按按钮的动作是“幂等”的。
或者按遥控器的静音键：排除两次按会出现取消的现象，按一次静音，再按多次，还是静音。状态不会叠加。

HTTP协议里面的get head put delete被设计为幂等， Post通常不是幂等，因为它常常用来新增资源，多次调用可能创建多条记录，有的业务post也需要做成幂等比如支付下单
## **记住常见的http状态码**

**2成3跳4客错5服错**
1xx：为信息，收到请求，继续处理
2xx：成功，你要的东西我处理好了
3xx：重定向，你得到别的地方去找
4xx：客户端错误，你的请求有问题
5xx：服务端错误，服务器自己崩了

200 ok 请求成功
201 created 已创建（注册）
204 no content 成功但无返回内容（删除）

301moved permanently 永久重定向（网站已经搬家了）
302 found 临时重定向（暂时的跳转，下次还来原地址问）
304 not modified 未修改（缓存有效，直接用本地缓存，不用再传数据）

400 bad request 错误的请求：请求语法错误，服务器看不懂
401 unauthorized 未认证：你没登录，请先提供身份凭证
403 forbidden 禁止访问：你登录了，但你这账号没权限看
404 not found 找不到：人人皆知，访问的资源不存在
405 method not allowed 方法不允许：get接口你却用post去调

500 服务器内部错误：代码通用报错
502 网关错误
503 服务不可用
504 网关超时

## **了解跨域，options+post**

CORS把跨域请求分为**简单请求**和**非简单请求**（需要预检的请求）。

当跨域 **POST** 的**Content-Type**是**application/json**或带有自定义头时，浏览器会先自动发一个**OPTIONS**请求进行“**探路**”（也叫做预检），服务器同意后才会发送**真正的 POST**。

Post有时候发options有时候不发，关键在于content type和自定义头部。因为绝大多数前后端分离项目都用application或者json传输数据，所以我们几乎总是看到options加Post的组合
